# 总体设计与当前实现的偏离评估
更新日期：2026-04-02

## 0. 文档目的

本文用于对照 [总体设计-latest.md](./总体设计-latest.md)，评估当前仓库实现与原始目标之间的偏离程度。

本文不以“是否已经做出可演示系统”为评判标准，而以如下问题为判断依据：

- 当前实现是否已经落到总体设计所定义的目标形态
- 当前实现为了做出纵切验证，在哪些地方主动压缩了架构
- 这些偏离哪些是阶段性可接受的，哪些如果长期保留会伤害总体目标
- 当前系统对外最诚实的表述应当是什么

本文刻意避免美化性表述。若某处尚未达到总体设计要求，将直接指出。

---

## 1. 结论先行

结论很明确：

> 当前系统没有偏离总体设计的主方向，但与总体设计所描述的完整目标形态相比，存在显著的阶段性压缩实现。

更准确地说：

- **方向上没有跑偏**
  - 系统仍然坚持“LLM 负责高不确定性语义，控制面持有治理权，执行面负责真实运行，结果必须结构化与可审计”这一核心原则
- **实现形态上明显收缩**
  - 当前更像一个围绕 `water_yield` 的受治理验证型纵切，而不是总体设计中那套完整的 `LLM + Skills + MCP + Metadata Catalog + 六层协同` 工程系统
- **风险不在于做错了，而在于说多了**
  - 如果后续对外把当前实现表述成“总体设计已基本落地”，那会失真

因此，当前最诚实的系统定位应是：

> 当前实现已经证明了“自然语言 -> 受治理 contract -> 真实 InVEST 执行 -> 结构化结果与审计”这条验证型主链可以成立，但尚未完整实现总体设计中面向生态问题智能分析的全量架构目标。

---

## 2. 原始目标是什么

根据 [总体设计-latest.md](./总体设计-latest.md)，原始目标不是“做一个能跑的单案例系统”，而是构建一套：

- 以 `Skills` 组织领域认知
- 以 `MCP` 组织能力执行
- 以 `LLM` 承担高不确定性语义职责
- 以控制面掌握唯一治理权
- 以规划面承担确定性编排
- 以执行面承接异步重计算
- 以数据面维护状态、审计、目录化元数据与工件

共同组成的六层协同工程系统。

该目标隐含了几个强要求：

1. **Skill-first 不是口号，而是资产化要求**
   - Skill 应以独立资产形式存在，而不是把知识散落在代码中
2. **MCP 不是工具调用代名词，而是标准能力面**
   - inspect / validate / job control / summarize 应成为受控能力暴露层
3. **Metadata Catalog 是认知面与规划面的默认事实源**
   - 恢复、绑定、校验、执行不应主要依赖临时附件元数据
4. **认知面与规划面必须在逻辑上有稳定边界**
   - 认知负责语义，规划负责编排，控制面负责治理
5. **系统目标是面向生态问题分析的一般化方法，不是单一模型接通**

如果按这个标准来看，当前实现与原始目标之间存在明显差距。

---

## 3. 当前实现的真实位置

当前仓库实现已经具备的能力，主要是：

- `goal-route / passb / repair proposal / final explanation` 已可由 LLM 参与
- `WAITING_USER / resume / cancel / audit / artifact / workspace` 已形成治理闭环
- 真实 InVEST `water_yield` 案例已经可执行
- clarify 与 `case_projection` 已进入主链
- 前端已能展示 task detail / result / artifact / LLM trace

这说明当前系统已经不是：

- 纯 demo
- 纯规则直连
- 纯前端包装

但它也还不是：

- 多 skill 的生态分析系统
- 完整的 Skill 资产化系统
- 完整的 MCP 能力面系统
- 目录化元数据驱动系统
- 总体设计所描述的成熟六层落地形态

因此，当前最合理的位置判断是：

> 当前实现属于“验证型纵切 + 强治理主链 + 单 capability 深化”，而不是“总体设计全量实现”。

---

## 4. 主要偏离点

下面按偏离重要性排序，直接评估当前实现与总体设计的差距。

### 4.1 偏离一：系统目标被收缩成单一 capability 纵切

#### 原始目标

总体设计的目标是“面向生态问题智能分析的 LLM + Skills + MCP + InVEST 协同系统”，强调的是方法体系和一般化能力。

#### 当前状态

当前真正落地并贯穿主链的能力基本仍围绕单一 capability：

- `water_yield`

虽然已有 clarify 和 case projection，但主能力空间仍然非常窄。

#### 判断

- **偏离等级：高**
- **性质：阶段性收缩**

#### 风险

如果长期停留在这个状态，系统会从“通用架构方案”退化为“单模型特化工程”。

#### 当前是否可接受

- **短期可接受**
  - 作为 Phase0/1/3 的验证型纵切，这是合理策略
- **长期不可接受**
  - 若继续宣称总体设计正在落地，则会失真

---

### 4.2 偏离二：Skill-first 目标尚未真正资产化

#### 原始目标

总体设计明确要求每个 Skill 至少应包含：

- `SKILL.md`
- `skill_profile.yaml`
- `model_mapping.yaml`
- `parameter_schema.yaml`
- `validation_policy.yaml`
- `repair_policy.yaml`
- `interpretation_guide.yaml`
- `plan_templates.yaml`
- `mcp_tools_map.yaml`

这意味着领域知识、模型映射、参数要求和解释规范应尽量以 Skill 资产存在。

#### 当前状态

当前仓库中，这套 Skill 资产并未真实落地为运行时资产。
核心知识主要仍写在代码中，例如：

- `Service/planning-pass1/app/skill_catalog.py`
- `Service/planning-pass1/app/case_registry.py`

搜索结果表明，上述 YAML 资产主要存在于文档中，而不是系统的真实运行边界。

#### 判断

- **偏离等级：高**
- **性质：架构目标未落地**

#### 风险

- 知识仍然散落在代码里
- 版本化、审计和跨 capability 扩展会越来越困难
- 后续新增 capability 容易继续走“代码堆叠”而不是“资产接入”

#### 当前是否可接受

- **短期可接受**
  - 在单纵切阶段，代码内置知识可以换取更快闭环
- **长期不可接受**
  - 这与总体设计中的“知识配置化”原则直接冲突

---

### 4.3 偏离三：MCP 能力面并未真正形成

#### 原始目标

总体设计将 MCP 能力面定义为：

- inspect
- validate
- job control
- summarize / extract

的标准化暴露层。

#### 当前状态

当前实际实现更像是一个统一 Python service，同时提供：

- `/cognition/goal-route`
- `/planning/pass1`
- `/cognition/passb`
- `/validate/primitive`
- `/planning/pass2`
- `/repair/proposal`
- `/cognition/final-explanation`
- `/jobs`

这是一种受控 HTTP service 结构，但不是总体设计里那种明确的 MCP 能力组织层。

#### 判断

- **偏离等级：高**
- **性质：能力面压缩**

#### 风险

- “MCP” 会沦为概念，而不是真实架构层
- inspect / validate / summarize 无法形成统一的能力约束与扩展边界
- 技术债会集中到一个过渡 service 中

#### 当前是否可接受

- **短期可接受**
  - 作为纵切验证，这种压缩能减少联调复杂度
- **长期不可接受**
  - 若后续继续扩 capability，service 会变成多职责耦合中心

---

### 4.4 偏离四：Metadata Catalog 尚未真正成为默认事实源

#### 原始目标

总体设计要求存在目录化元数据服务，并明确提出：

- `asset_id`
- `asset_version`
- `hash`
- `parse_status`
- `MIN_READY`
- `FULL_READY`
- `delta_inventory`

应成为认知面、规划面和恢复链的基础事实。

#### 当前状态

当前系统虽然已经实现了 `MIN_READY` 风格的恢复门槛，但实际判断主要仍依赖附件元数据可见性，例如：

- attachment 的 `id`
- `logical_slot`
- `file_name`
- `size_bytes`
- `stored_path`
- `checksum`

这与“完整 Metadata Catalog 驱动”仍有明显距离。

#### 判断

- **偏离等级：高**
- **性质：事实源简化**

#### 风险

- 当前恢复链能工作，但尚不足以支撑更复杂的资产治理
- asset-level 版本、目录事实与物理文件一致性尚未真正成为统一基础
- 后续多 case、多 capability、多租户扩展会受限

#### 当前是否可接受

- **短期可接受**
  - 对当前 repair/resume 闭环已足够
- **长期不可接受**
  - 这与总体设计第 11 节的元数据底座目标不一致

---

### 4.5 偏离五：认知面、规划面、执行面仍被压缩在单一服务中

#### 原始目标

总体设计强调六层协同，每一层有明确职责和边界。

#### 当前状态

在逻辑职责上，当前已经开始区分：

- cognition
- planning
- validate
- repair
- jobs

但在物理形态上，它们仍主要压缩在同一个 Python service 中。

#### 判断

- **偏离等级：中**
- **性质：物理部署压缩**

#### 风险

- 逻辑边界仍可能被实现细节侵蚀
- 后续若服务继续膨胀，会让边界名义存在、实现却混杂

#### 当前是否可接受

- **短期可接受**
  - 这是非常常见的纵切策略
- **长期可接受吗**
  - 只在逻辑边界持续收紧的前提下，才勉强可接受
  - 若后续 prompt、planning、runtime 逻辑继续在一个 service 中互相渗透，则不可接受

---

### 4.6 偏离六：控制层仍残留语义和路由硬编码

#### 原始目标

总体设计明确要求：

- 控制面不负责 GIS 领域语义
- 控制面不负责模型选择知识
- 控制面不负责参数语义推理

#### 当前状态

Java backend 中仍可见明显的语义/路由硬编码，例如：

- 基于 query 关键词推断 primary skill
- 基于 query 判断 real-case 偏好
- 在控制层中做部分 real-case execution preference 注入

这说明控制层还没有完全退回“纯治理主机”的位置。

#### 判断

- **偏离等级：中到高**
- **性质：过渡期越层**

#### 风险

- 领域知识会继续沉积到控制层
- 后续认知层抽离会越来越痛苦
- 与总体设计中的“认知面负责语义，控制面只负责治理”原则冲突

#### 当前是否可接受

- **短期勉强可接受**
  - 如果明确视为 transitional logic
- **长期不可接受**
  - 这类硬编码若不清理，会直接侵蚀总体设计的分层原则

---

## 5. 没有明显偏离，甚至做对了的地方

尽管存在上述偏离，但以下方面与总体设计是对齐的，甚至可以认为做得比较扎实：

### 5.1 控制层仍掌握主要 authority

当前系统中，以下关键工作流 authority 仍主要由控制层掌握：

- `WAITING_USER`
- `can_resume`
- `required_user_actions`
- task state transitions
- repair dispatch

这与总体设计和 AGENTS 边界是一致的。

### 5.2 没有让 LLM 越权成为执行或治理 authority

当前 LLM 参与的主要是：

- goal parse
- route / passb
- repair proposal
- final explanation

它没有直接决定：

- 是否提交 job
- 是否允许 resume
- workflow terminal state
- execution truth

这点非常重要，也说明方向没有跑偏。

### 5.3 执行链已经真实可运行，而不是伪闭环

当前系统已经证明：

- real provider 可运行
- result / artifact / audit 可落盘
- runtime evidence 可展示

这意味着当前不是“架构 PPT 原型”，而是真正有执行闭环的工程系统。

### 5.4 clarify 与 case projection 的引入方向是正确的

这条线虽然还不成熟，但它与总体设计中的“受治理认知入口”方向是一致的。

---

## 6. 偏离的本质是什么

当前偏离的本质，不是“做错了体系”，而是：

> 为了先证明真实执行闭环可成立，系统主动把总体设计压缩成了一条受治理的验证型纵切。

这种压缩主要体现在：

- 从通用生态分析系统压缩成单 capability 纵切
- 从 Skill 资产化压缩成代码内置 catalog
- 从 MCP 能力面压缩成单 service API
- 从 Metadata Catalog 压缩成 attachment-driven ready facts
- 从六层物理协同压缩成多层逻辑边界共存于少量服务

因此，当前系统不是“背离总体设计”，而是：

> 只实现了总体设计中最靠近真实价值闭环的那一段。

这既是现实选择，也是风险来源。

---

## 7. 哪些偏离可以接受，哪些不能长期接受

### 7.1 可以接受的偏离

以下偏离在当前阶段可以接受：

- 单 capability 深化，而非多 skill 扩展
- 认知/规划/修复逻辑暂时共存于单一 Python service
- 前端先做验收与 trace 可视化，而非产品化重构
- 第二层次的资产治理、目录化元数据治理暂时简化

这些都属于典型的“先验证主价值链”策略。

### 7.2 不能长期接受的偏离

以下偏离如果继续保留，就会真正破坏总体设计目标：

- Skill 资产长期不外置
- 控制层长期保留领域语义和路由硬编码
- MCP 长期停留在概念层，而不是能力边界层
- Metadata Catalog 长期缺位
- 多 case / clarify 成立后仍缺乏 case space 证据

这些不是阶段性压缩，而会变成结构性债务。

---

## 8. 当前最诚实的系统表述

如果现在要给导师、团队、论文或答辩使用一句不失真的表述，建议采用：

> 当前实现已经证明，系统能够在受治理控制下完成“自然语言请求 -> 结构化认知投影 -> 真实 InVEST 执行 -> 结构化结果与审计”的验证型闭环；但它仍是围绕 `water_yield` 能力构建的纵切实现，尚未完整落地总体设计中以 Skills、MCP 和 Metadata Catalog 为基础的全量架构目标。

这句话有三个好处：

- 承认成绩
- 不夸大阶段
- 给后续演进留出清晰空间

---

## 9. 下一步最重要的不是“补齐文档”，而是补齐哪种偏离

如果从总体设计出发，只能选一个优先修正方向，那么最值得优先收口的不是前端，也不是把六层拆成更多服务，而是：

> 把“单 case 可执行成功链”推进为“至少两个真实可执行 case 下可验证的 governed projection 与 clarify 能力”。

原因是：

- 这一步能直接验证当前认知/治理主链是否真的进入了 case space
- 它会倒逼 registry、clarify、resume、runtime evidence 一致性变硬
- 它比“先做 MCP 外形”更能证明体系不是单样例包装

在此之后，最该收口的两条债务是：

1. Skill 资产化
2. Metadata Catalog 真实化

这两条如果不补，总体设计长期仍然只会停留在设计层。

---

## 10. 最终结论

最终结论可以压缩成三句话：

1. **没有偏离总体设计的主方向。**
2. **但当前实现与总体设计的完整目标相比，仍存在显著的阶段性压缩。**
3. **当前最大的风险不是技术错误，而是把“验证型纵切”误表述成“总体架构已落地”。**

因此，当前最合理的判断应是：

> 这是一个方向正确、闭环已成立、但仍明显早于总体架构落地完成度的系统。
