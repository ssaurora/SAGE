# 真实案例-智能调用过程说明

## 1. 文档目的

本文基于已经成功跑通的一条真实任务，说明当前系统如何完成一次真实的 InVEST 智能调用，并明确回答两个问题：

1. 用户问题是如何被系统逐层处理的。
2. 当前到底哪些环节用了大模型，哪些环节还没有用。

这份说明基于真实成功任务：

- `task_id`: `task_20260328033907386_a255eb47`
- `job_id`: `job_20260328033907531423_e5ced28f`
- `provider_key`: `planning-pass1-invest-local`
- `runtime_profile`: `docker-invest-real`
- `case_id`: `annual_water_yield_gura`
- 最终状态：`SUCCEEDED`

用户原始问题是：

```text
run a real case invest annual water yield analysis for gura
```

## 2. 先给结论

这条真实成功案例里，用户问题并**没有**直接送给大模型来做主规划。

当前真实成功链路是：

- 前端收集自然语言问题
- Java backend 用规则路由把问题识别为 `water_yield` 真实案例
- backend 调用 planning service 的 `pass1 / passb / validation / pass2`
- 这些 planning 接口当前仍然是 deterministic 规则实现，不是 LLM 推理
- execution 层把结构化 contract 交给真实 `natcap.invest.annual_water_yield.execute(args)`
- runtime 产出真实 output files 和 output registry
- backend 再把 runtime facts 投影成 result bundle / artifact catalog / final explanation

当前真正会把内容送给大模型的环节，是 **repair proposal**，也就是任务进入 `WAITING_USER` 之后的用户提示文案生成；这条成功案例没有进入该分支，所以本次成功运行中没有发生大模型调用。

一句话压缩：

> 这次成功案例证明了“真实 InVEST 调用闭环”已经成立，但还没有证明“主规划链已经由大模型驱动”。

## 3. 这条真实任务按六层架构是怎么走的

### 3.1 Presentation Layer

用户在前端提交自然语言问题：

```text
run a real case invest annual water yield analysis for gura
```

前端本身不做业务推理，只把问题发送给 backend。

### 3.2 Control Layer

backend 负责：

- 接收任务创建请求
- 生成 `goal_parse` 和 `skill_route`
- 调用 planning service
- 持久化 `pass1_result_json / passb_result_json / validation_summary_json / pass2_result_json`
- 创建 job
- 监听 job 结果并做 task state transition
- 把 runtime facts 投影到 task detail / result / artifacts

这条任务的 control-layer 结论是：

- `goal_type = water_yield_analysis`
- `execution_mode = real_case_validation`
- `provider_preference = planning-pass1-invest-local`
- `runtime_profile_preference = docker-invest-real`

### 3.3 Cognition Layer

当前主成功链里没有真正启用 LLM cognition。

目前 `cognition/passb` 这个接口名字虽然叫 cognition，但实现仍是 deterministic 规则编排，不是 prompt + model 推理。

当前 LLM 只在 repair proposal 分支中使用，见第 7 节。

### 3.4 Capability Layer

`water_yield` skill 定义了：

- capability key
- required / optional roles
- slot schema
- role 到 arg 的映射
- output contract
- validation hints

它决定“这个能力需要什么输入、输出什么工件”，但不决定任务状态。

### 3.5 Execution Layer

真实执行层使用：

- `provider_key = planning-pass1-invest-local`
- `runtime_mode = invest_real_runner`
- `contract_mode = invest_real_case_v1`

最终调用的是：

```python
natcap.invest.annual_water_yield.execute(args)
```

这一步已经不再是 deterministic stub。

### 3.6 Data Layer

数据层保存了这次任务的关键事实：

- `task_state`
- `analysis_manifest`
- `job_record`
- `runtime_request.json`
- `runtime_result.json`
- `run_manifest.json`
- `result_bundle.json`
- `artifact_catalog`

这些数据让整条调用链可追溯。

## 4. 用户问题在成功案例中是如何被逐层处理的

## 4.1 第一步：用户原始问题进入 control layer

原始问题：

```text
run a real case invest annual water yield analysis for gura
```

backend 的 `GoalRouteService` 对这个问题做规则解析，得到：

```json
{
  "goal_type": "water_yield_analysis",
  "analysis_kind": "water_yield",
  "execution_mode": "real_case_validation",
  "provider_preference": "planning-pass1-invest-local",
  "runtime_profile_preference": "docker-invest-real"
}
```

这里没有调用大模型。

## 4.2 第二步：backend 调用 planning/pass1

backend 调用 planning service：

```text
POST /planning/pass1
```

传入的是结构化请求，大致包含：

```json
{
  "task_id": "task_20260328033907386_a255eb47",
  "user_query": "run a real case invest annual water yield analysis for gura",
  "state_version": 1
}
```

`pass1` 返回的是能力事实，不是模型生成文本。关键内容包括：

- `capability_key = water_yield`
- `selected_template = water_yield_v1`
- required roles: `precipitation`, `eto`
- optional roles: `depth_to_root_restricting_layer`, `plant_available_water_content`
- role arg mappings
- slot schema
- graph skeleton

这一步的作用是把“自然语言请求”压缩成“这个任务在能力层上属于哪个 skill，以及这个 skill 的输入输出契约是什么”。

## 4.3 第三步：backend 调用 cognition/passb

backend 调用：

```text
POST /cognition/passb
```

虽然 endpoint 名叫 `cognition/passb`，但当前实现是 deterministic 规则逻辑。它读取：

- 用户原始 query
- `pass1_result`
- 内置 case contract

然后输出 candidate 结构。

## 4.4 第四步：backend 调用 validate/primitive

backend 调用：

`pass2` 产生真正可执行的最小编译产物：

## 5. 这条成功案例里，大模型到底有没有参与

答案是：

**没有参与主成功链。**

更准确地说：

- 用户自然语言 query 没有直接发给大模型
- `GoalRouteService` 是规则路由
- `pass1 / passb / validation / pass2` 当前都是 deterministic 编排
- `final_explanation` 也是根据 runtime facts 规则生成
- 真正执行模型的是 `natcap.invest`，不是 LLM

所以如果问：

> “这次真实 InVEST 成功案例里，大模型输出了什么？”

当前诚实答案是：

> 这次成功案例里，大模型没有被调用，因此也不存在“大模型输出主规划结果”的事实。

## 6. 那当前系统里，问题是怎么给到大模型的

当前系统里，大模型入口在 **repair proposal** 分支，而不是主规划分支。

它只在任务进入 `WAITING_USER` 时参与，作用是：

- 用结构化事实生成更友好的用户解释
- 生成 `resume_hint`
- 生成 `action_explanations`

但它**不决定**状态机，也不决定 `WAITING_USER` / `FAILED` / `can_resume`。

这些仍由 backend dispatcher 决定。

## 7. repair proposal 分支中，大模型收到什么、输出什么、输出后如何处理

## 7.1 什么时候会调大模型

当任务进入 `WAITING_USER`，backend 会构造：

- `waiting_context`
- `validation_summary`
- `failure_summary`
- `user_note`

然后调用 service：

```text
POST /repair/proposal
```

## 7.2 送给大模型的输入长什么样

service 中 `SAGE_REPAIR_PROVIDER=glm` 时，会调用 GLM。

system prompt 是固定的，要求模型：

```text
You are a repair assistant.
Return ONLY strict JSON with keys:
user_facing_reason (string), resume_hint (string),
action_explanations (list of {key,message}), notes (list of strings).
Do not include extra keys.
```

user message 不是自然语言原文，而是结构化 JSON，包含：

```json
{
  "waiting_context": {
    "waiting_reason_type": "...",
    "missing_slots": [
      {"slot_name": "...", "expected_type": "...", "required": true}
    ],
    "invalid_bindings": [],
    "required_user_actions": [
      {"action_type": "upload", "key": "...", "label": "...", "required": true}
    ],
    "resume_hint": "...",
    "can_resume": false
  },
  "validation_summary": {
    "is_valid": false,
    "missing_roles": [],
    "missing_params": [],
    "error_code": "...",
    "invalid_bindings": []
  },
  "failure_summary": {
    "failure_code": "...",
    "failure_message": "...",
    "created_at": "..."
  },
  "user_note": "..."
}
```

关键点：

- backend 不把 workflow authority 交给模型
- 送给模型的是已经定型的结构化事实
- 模型只负责把这些事实转成更友好的 repair 文案

## 7.3 期望大模型输出什么

期望模型返回严格 JSON：

```json
{
  "user_facing_reason": "...",
  "resume_hint": "...",
  "action_explanations": [
    {"key": "...", "message": "..."}
  ],
  "notes": ["..."]
}
```

## 7.4 大模型输出之后怎么处理

处理流程是：

1. service 解析模型输出 JSON
2. service 组装成 `RepairProposalResponse`
3. backend `RepairProposalService` 校验 schema
4. 如果字段缺失、JSON 非法、请求失败，直接 fallback 到 deterministic repair proposal
5. 最终 repair proposal 会被持久化为 advisory facts
6. 但 task workflow state 仍由 dispatcher / control layer 决定

所以这里的大模型是：

- `advisory`
- `schema constrained`
- `can fail safely`
- `not workflow authority`

## 8. 这份真实案例证明了什么

这次真实 Gura Annual Water Yield 案例已经证明：

1. 当前系统已经可以把自然语言请求落成真实 InVEST 的执行 contract。
2. 真实输入路径、provider 选择、runtime profile、artifact promotion 都已经能闭环。
3. 主链成功并不依赖大模型，说明治理主线和真实执行主线已经分清。
4. 当前大模型的实际职责主要仍是 repair explanation，而不是主规划 authority。

## 9. 当前最重要的认知结论

如果你要用一句话向团队解释当前系统状态，最准确的说法是：

> 我们已经跑通了“真实 InVEST 执行闭环”，但当前成功链还是“规则规划 + 真实模型执行”；大模型目前主要用于 repair proposal，而不是主规划裁决。

这不是缺点，而是当前阶段的真实状态。

它意味着：

- 架构已经足以承载真实模型
- 但如果后续要证明“LLM 驱动的智能规划也成立”，还需要单独推进下一阶段
- 不能把当前成功案例误说成“主规划已经由大模型完成”
