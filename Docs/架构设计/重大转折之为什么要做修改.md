## 一、背景

`SAGE` 当前已经不是一个简单的“LLM + runtime”的原型系统，而是一个具备真实治理能力的分析执行系统。
当前主链已经明确存在：

* 用户从自然语言发起分析请求
* 系统进入 governed planning
* 当输入不足时，任务进入 `WAITING_USER`
* 用户补充附件或 override 后，调用 `/resume`
* 系统重新进入 validation / dispatch
* 最终产生可追溯的 result / artifact / audit / runtime evidence 

同时，现有系统已经拥有：

* `POST /tasks`
* task detail / events / stream / manifest / result / runs / artifacts / audit / catalog / contract 等读接口
* repair/resume 主路径
* workspace / result bundle / artifact index / run history / runtime evidence 等 traceability 能力
* catalog-first 与 contract-first 的治理边界 

也就是说，**SAGE 的治理底座已经成立**。

---

## 二、当前问题不在底座，而在主交互模型

虽然底座已经成立，但当前系统的用户主交互模型仍然是：

> 自然语言创建一个 task → 进入 task detail / 等待 / 结果等页面

这条路径对后端治理系统是成立的，但对未来真正的产品形态还不够。

原因在于，用户真正的使用方式并不是：

* 理解 task 状态机
* 主动操作治理页
* 学习输入槽位和底层模型配置
* 把 workbench 当成主页面

用户真正想做的是：

* 提出分析问题
* 补一点自己手上的数据
* 回答系统必要的追问
* 在必要时确认、澄清或补数据
* 让系统替自己把底层分析任务推进到底
* 最后拿到可解释、可追溯的结果

也就是说，**用户主体验应该是持续协作，而不是 task detail 浏览**。

---

## 三、当前设计和真实产品目标之间的核心落差

### 1. 当前系统的“第一正式对象”是 task，而不是用户协作过程

现在用户一进入系统，核心对象就是 task。
task 非常适合作为治理与执行对象，但它并不天然等于“用户此次分析协作的全过程”。

一个 task 更像：

* 受治理的执行单元
* 带状态机的控制对象
* 可冻结、可审计、可恢复的运行对象

但用户感知到的并不是“我在操作一个 task”，而是：

> “我正在和系统围绕一个分析目标持续协作”

这层对象在当前系统里还没有被正式建模。

---

### 2. 自然语言目前更像“任务创建入口”，而不是“主交互通道”

当前系统已经从自然语言开始，这很好。
但自然语言的角色仍然偏向“启动流程”。

用户提完一次自然语言请求后，系统很快转入 task/detail/waiting/result 语义。
这导致自然语言没有贯穿整个协作周期。

而真实产品里，用户在任务运行过程中仍然会需要：

* 继续补充说明
* 回答 clarify 问题
* 解释上传内容
* 让系统解释为什么被阻塞
* 在结果出来后继续追问、要求重跑或比较方案

这说明：
**自然语言不应该只存在于创建时，而应该贯穿任务生命周期。**

---

### 3. 当前 read model 更强于“任务治理消费”，弱于“会话协作消费”

目前 `SAGE` 的 detail / manifest / result / audit / catalog / contract 等读模型已经比较强，这很好 。
但这些读模型主要服务于：

* 任务详情
* 治理事实
* 执行结果
* 审计与追溯

它们还没有形成一个真正面向“持续会话协作”的上层视图。

目前缺的不是底层事实，而是**面向用户持续协作的聚合视图**。

---

### 4. 如果继续只围绕 task 扩展，前端会越来越重，语义也会越来越别扭

如果不做这次修改，后续前端为了实现“持续协作”体验，只能被迫：

* 自己把 waiting_context 解释成对话
* 自己把 events 解释成消息流
* 自己从 task / result / manifest / audit 里拼会话上下文
* 自己发明前端层的 conversation 语义

这样会导致：

* 前端越来越像 workflow brain
* 后端事实与前端语义分裂
* mock 与 real object 持续漂移
* 页面越来越重，跨页面跳转越来越复杂

这与 `SAGE` 当前明确坚持的 layered ownership 是相冲突的：
后端拥有 workflow、state、freezing、audit、orchestration，frontend 只负责 render and interact 

---

## 四、为什么是现在改，而不是以后再说

现在是一个很合适的时点，因为：

### 1. 底座已经够强，可以承接上层重构

现在不是在 Phase0 的薄壳阶段了。
你已经有：

* governed main chain
* WAITING_USER / resume
* traceability
* manifest / result / audit / catalog / contract
* thinner orchestration and unified read models 

这意味着：
**现在补上层对象，不需要推翻底座。**

### 2. 产品主入口还没完全定型

现在前端仍在重构阶段，`sage-web` 也还处于静态原型和产品壳阶段。
如果现在不把后端交互模型补齐，未来前端会被迫用脆弱映射去顶替正式对象。

### 3. 越晚改，代价越大

如果等到前端已经深度围绕 task detail 和 workbench 固化，再回头加 session/message 层，会导致：

* API 和读模型再拆一遍
* 前端页面重排一次
* 语义映射再改一次

现在改，成本最小，收益最大。

---

## 五、本次修改的本质

这次修改不是推翻 `SAGE`。
这次修改的本质是：

> **在现有 task/governance/execution 底座之上，补一层“面向持续协作的会话对象与读模型”。**

因此，修改的目标不是弱化治理，
而是让治理能力更自然地被用户主交互消费。

---

## 六、结论

我们现在要做修改，不是因为 `SAGE` 现有系统做错了，
而是因为它已经完成了“治理型任务系统”这一阶段，现在必须进入下一阶段：

> **从“自然语言创建 task 的治理系统”，升级成“以持续协作为主、以治理任务为底座的分析系统”。**

这次修改的必要性在于：

1. 用户主体验是持续协作，不是 task detail 浏览
2. 当前缺少 session/message 这一层正式对象
3. 当前 read model 更偏治理消费，弱于会话消费
4. 现在改动成本低，且底座已经足够成熟
5. 如果不改，前端会被迫发明语义，破坏前后端边界
