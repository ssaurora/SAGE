## 阶段 0：可演示内核（2~3 周思路）

这一阶段的目标不是“架构成立”，而是**从一句自然语言到一个真实结果的最小闭环**。  
只支持：

- 单 Skill：`water_yield`
    
- 单模板
    
- 单 provider
    
- 单执行路径
    
- 单租户、低并发
    
- 不做恢复事务
    
- 不做模板协商
    
- 不做复杂预处理编排
    
- 不做多轮自动修复，只做有限轮次提示与重试
    

这和总体设计中“单 Skill、单主题最小闭环”的建议是一致的。

### 这一阶段要保留的核心能力

认知面只保留最小链路：Intent Parser → Skill Router → Context Enricher → Parameter Builder → Decision Summary / Explanation。总体设计已经把这些节点列出来了，完全可以先做最少集。

规划面只做最小两程编译中的“弱化版”：

- Pass 1：选唯一模板，产出 `slot_schema_view`、`graph_skeleton`
    
- Pass 2：结合绑定结果产出 `materialized_execution_graph` 和基础 `runtime_assertions`  
    不做图重写，不做模板协商，不做复杂 canonicalize。
    

控制面只做最小主干状态机：

- `CREATED`
    
- `COGNIZING`
    
- `PLANNING`
    
- `VALIDATING`
    
- `QUEUED`
    
- `RUNNING`
    
- `SUCCEEDED / FAILED`
    

并且先把 `TaskState`、`state_version`、基础事件流建起来。控制面的最小工程落地顺序也明确把这一步放在第一位。

能力面只做四类基础 contract：

- inspect
    
- validate
    
- job control
    
- summarize/extract  
    这是总体设计里已经明确的最小能力集合。
    

执行面只做最基础 job runtime：

- `ACCEPTED/QUEUED/RUNNING/SUCCEEDED/FAILED/CANCELLED`
    
- worker dispatch
    
- heartbeat  
    这正是执行面文档第一阶段定义的骨架。
    

数据面只建四张最关键表：

- `task_state`
    
- `analysis_manifest`
    
- `event_log`
    
- `audit_record`  
    数据面文档把它定义为第一阶段核心表，这很合理。
    

### 这个阶段交付什么

你应当能跑通这样一条链：

用户自然语言提问 → 识别为 `water_yield` → 绑定输入 → 生成参数草案 → 校验 → 生成最小执行图 → 提交异步 job → 执行 InVEST → 返回结构化结果对象 → 生成受约束解释。  
这条链正对应总体设计里的完整目标闭环。

### 这个阶段故意不做什么

不做 `/resume`、不做 `MIN_READY`、不做 `STATE_CORRUPTED`、不做 Artifact Promotion、不中间插入复杂 preprocess DAG、不做 provider version 治理。这些都很重要，但在这个阶段只会拖慢出样。控制面、能力面、数据面的文档都把它们放在后续步骤，顺序上也是支持这种收缩的。

## 阶段 1：受控可用版（先把系统“做稳”）

当阶段 0 已经能稳定演示后，下一步不是急着扩 Skill，而是把**治理闭环补齐**。这一阶段的目标是：系统不只是能跑一次，而是**能被重复使用、能处理常见错误、能安全取消、能留下证据**。

### 这一阶段新增的重点

首先补控制面的跨面编排和修复分流。控制面文档里已经定义了 Repair Dispatcher：参数/绑定类错误回认知面，图结构与预处理链类错误回规划面，元数据未就绪进入等待态，科学降级进入确认态。这个机制应该在这一阶段正式引入。

其次补执行面的 workspace 生命周期：

- create
    
- cleanup
    
- archive
    
- demolish
    
- run-scoped directory layout  
    这也是执行面第二阶段的重点。
    

再补 runtime assertion push-down，让执行面实际消费规划面生成的断言，而不是只靠前置校验。执行面把这一项列在第四阶段，但在你的场景里可以适当前提，因为它对稳定性很关键。

能力面则开始从“能调通”转向“有契约”：

- capability registry
    
- provider registry
    
- schema 固定
    
- 异步 contract 明确  
    能力面文档明确要求长时能力必须采用异步契约，不能在网关层同步阻塞。
    

数据面要补第二阶段的执行相关表：

- `job_record`
    
- `workspace_registry`
    
- `result_bundle_record`
    
- `artifact_index`
    
- `queue_record`  
    这样控制面和执行面才真正闭环。
    

### 这一阶段的标志

到了这一步，你的系统应当具备：

- 失败后能看见结构化错误
    
- 任务取消后能真正停掉作业
    
- 结果和关键文件能形成 `result bundle`
    
- 重复提交不会把状态冲乱
    
- 前端能看到任务流、事件流、结果和解释
    

这时它已经不是 demo，而是一个可供内部试用的系统。

## 阶段 2：强治理版（进入“系统”而不是“产品原型”）

这一阶段才开始补你文档里最有工业味道的内容：**恢复事务、等待语义、冻结、版本域解耦、状态损坏治理**。这部分很关键，但绝对不适合放在最前面做。

### 核心建设点

控制面进入第三、第四、第五步：

- `MIN_READY`
    
- `/resume`
    
- Ack / Commit / Rollback 恢复事务
    
- `PRE_PROCESSING`
    
- Feasibility Gate
    
- Artifact Promotion
    
- 终局闭环
    
- 全链审计与事件回放  
    这些都在控制面的最小工程落地顺序里写得很清楚。
    

其中最值得单独强调的是：**恢复必须是事务，不是“重新点一下运行”**。总体设计和控制面文档都明确说了，恢复前必须先完成新输入合并、最小元数据就绪、失效范围分析和状态版本校验，认知面返回 Ack 之后控制面才 Commit。

规划面在这一阶段进入真正的“工业编译器”形态：

- Static Graph Validator
    
- Graph Rewriter
    
- Canonicalizer
    
- Runtime Assertion Builder
    
- Graph Digest Builder
    
- Planning Summary Builder  
    也就是从“能生成图”升级为“能修图、能验图、能摘要图、能固定版本图”。规划面定位本来就是工业级确定性编译器，这些内容应该放在第二阶段以后实现。
    

数据面要补恢复与确认表：

- `confirmation_request`
    
- `checkpoint_alignment_record`  
    这正对应控制面的恢复事务和人工确认。
    

### 这一阶段完成后

你的系统开始具备这些能力：

- 用户补传数据后可恢复
    
- 系统能识别哪些对象失效、哪些对象可复用
    
- 出现跨层不一致时可进入 `STATE_CORRUPTED`
    
- 成功态和失败态都能被归档、审计和回放
    

到这里，它才真正有“工业级治理内核”的味道。

## 阶段 3：工业级扩展版（广度、隔离、版本治理、生态接入）

这一阶段的重点不再是补单链能力，而是让系统进入**多能力、多模板、多 provider、可演进**状态。

### 主要扩展方向

第一，扩 Skill 和模板库。  
从单 `water_yield` 扩到多个 InVEST 主题，但仍然坚持 Skill-first，不允许“用户一句话直接撞工具”。总体设计把这个原则写得很清楚。

第二，扩 provider 生态，但保持能力面“薄”。  
能力面文档明确要求走 registry、thin gateway、provider-first，而不是把能力面做成第二个大业务中心。到了这一阶段，Python worker、result extractor、artifact handler 等 provider 可以通过注册机制接入。

第三，补数据生命周期治理：

- heartbeat / lease / cancel token / rate limit 的 Redis keyspace
    
- bucket / prefix 约定
    
- retention lifecycle rules  
    数据面把这些放在第五阶段，是合理的，因为它们属于“规模化之后的治理”。
    

第四，增强执行面隔离：

- 容器 / Pod 级运行单元
    
- soft cancel / hard cancel
    
- 失败隔离
    
- partial rerun
    
- safe reuse whitelist  
    执行面文档已经把这些明确为后续阶段目标。
    

第五，增强解释与报告能力，但仍然坚持“先结构化结果，再 LLM 解释”。总体设计特别强调，不能让 LLM 直接看一堆路径猜结论，必须先有结构化结果对象。

---

## 我建议的真实实施顺序

如果把上面四阶段压成最实用的一条路线，我会这样排：

**第一期：跑通**

1. 单 Skill：`water_yield`
    
2. 单模板、单 provider
    
3. 最小控制面状态机
    
4. 最小认知链和两程编译弱化版
    
5. 异步 job runtime
    
6. 结构化结果 + 最小解释
    

**第二期：做稳**

1. Repair Dispatcher
    
2. workspace 生命周期
    
3. result bundle
    
4. cancel 真正落到物理运行单元
    
5. capability registry / provider registry
    
6. 执行与审计相关表
    

**第三期：做治理**

1. `MIN_READY`
    
2. `/resume`
    
3. Ack / Commit / Rollback
    
4. checkpoint/version 域解耦
    
5. graph rewrite / canonicalize / digest
    
6. confirmation / corrupted state / event replay
    

**第四期：做工业化**

1. 多 Skill / 多模板
    
2. provider 生态扩展
    
3. 容器化隔离与 partial rerun
    
4. retention / archive / destroy 策略
    
5. 全链审计、回放、复盘
    
6. 结果报告体系与前端治理界面

## 最关键的判断

你现在最容易犯的错误，是把“工业级设计”当成“第一期需求”。  
实际上，从你这些文档的落地顺序看，正确做法恰恰相反：

- **第一期只证明闭环**
    
- **第二期证明稳定**
    
- **第三期证明可恢复、可治理**
    
- **第四期才证明可扩展、可工业化**
    

这条路线和你的现有设计并不冲突，反而是最符合这些文档本身精神的。因为总体设计已经明确建议从单 Skill、单主题最小闭环起步；控制面、执行面、数据面、能力面也都分别给出了由简到繁的工程顺序。

下一步最合适的事情，不是再补一版架构文档，而是把这个四阶段路线直接落成一份**实施清单**：每阶段分别拆成前端、Spring Boot、Python Worker、数据库四条任务线。
