# 四周推进计划-结合当前代码的修订版
更新日期：2026-04-02

## 0. 文档目的

本文用于对“接下来 4 周的开发推进计划”做一次结合当前代码基线的批判性修订。

这份修订版不重复原计划中“方向正确”的部分，而是重点回答三个问题：

- 原计划哪些判断是准确的，应该保留
- 原计划哪些目标需要收紧，避免 4 周内做成口号式承诺
- 结合当前仓库代码，未来 4 周最合理、最诚实、最能收缩偏离的推进顺序是什么

本文的基线依据包括：

- [总体设计-latest.md](../架构设计/总体设计-latest.md)
- [总体设计与当前实现的偏离评估.md](../架构设计/总体设计与当前实现的偏离评估.md)
- 当前 Java backend、Python service 与前端实现状态

## 1. 总体判断

先给结论：

> 这份 4 周计划的方向是对的，而且比继续堆 UI、继续扩大“LLM 更自由”叙事要清醒得多；但若要把它变成一份真正可开工、可验收、可对外复述的计划，就必须按当前代码现实做一次收紧。

当前最值得保留的判断有四个：

- 优先解决“单 case 依赖”而不是继续打磨演示界面
- 把 Skill 从代码知识推进到资产知识
- 让 Metadata Catalog 从原则变成真实事实源
- 让控制层退出领域知识承担者角色，回到治理主机定位

但原计划中也有三处如果不修订，会变成高风险承诺：

1. “至少为 2 个 skill 补出最小资产包”这句话在当前代码基线上偏激进  
   目前系统真正贯穿主链的仍是单一 capability/skill：`water_yield`。当前更现实的目标应是：
   - 先完整资产化 `water_yield`
   - 再为未来第二 skill 预留资产接入骨架
   - 而不是在 4 周内为了满足措辞，制造一个并不成熟的“第二 skill”

2. “Metadata Catalog 成为默认事实源”方向正确，但当前代码离完整 Catalog 还很远  
   因此 4 周内更合理的目标应是：
   - 做出最小可消费 catalog slice
   - 让它成为认知面、规划面和 resume gate 的默认事实源
   - 而不是宣称 Catalog 平台本身已经完整落地

3. “能力面 contract 化”必须坚持 thin contract，而不能演变成新的厚平台重写  
   如果 Week 4 误走成“大网关 / 大平台 / 全 MCP 重建”，不仅 4 周做不完，还会把当前已经跑通的治理主链重新搅乱

因此，本文的核心修订口径是：

> 未来 4 周的目标，不是从“验证型纵切”一步跃迁到“总体设计完整落地”，而是把当前系统从“单 case、单 skill、代码知识、附件事实驱动”的状态，推进到“至少两个真实可执行 case、一个真实 skill 资产包、一个可消费的最小 catalog、一个更薄更诚实的控制层、以及一组真正被消费的 contract”。

## 2. 当前代码基线：哪些判断必须面对

这份修订必须建立在当前代码真实状态上，而不是建立在理想状态上。

### 2.1 当前只有一个真实可执行 case

当前 `water_yield` case registry 已存在，但真实可执行的 case 仍只有一个：

- `annual_water_yield_gura`：`executable=True`
- `annual_water_yield_blue_nile_fixture`：`executable=False`

证据见：

- [case_registry.py](/e:/paper_project/SAGE/Service/planning-pass1/app/case_registry.py)

这意味着：

- clarify / case projection 的结构已经形成
- 但“多 case governed projection 能力”还没有真正成立
- 原计划把“第二真实案例接入”放在第 1 周，是完全正确的

### 2.2 当前真正贯穿主链的仍是单一 skill/capability

当前真正通过主链落地的 canonical capability 仍是：

- `water_yield`

而且 skill 定义仍主要存在于代码中，不是资产中。

证据见：

- [skill_catalog.py](/e:/paper_project/SAGE/Service/planning-pass1/app/skill_catalog.py)

这意味着：

- “Skill 资产化”是必须做的
- 但原计划里“至少为 2 个 skill 补最小资产包”的说法，需要调整成更贴近当前主链现实的版本

### 2.3 当前 Metadata Catalog 还没有真正存在

当前系统虽然已经有 `MIN_READY` 风格的恢复门槛，但本质仍是基于附件元数据与 override 的就绪性判断，而不是目录化元数据事实源。

证据见：

- [MinReadyEvaluator.java](/e:/paper_project/SAGE/BackEnd/src/main/java/com/sage/backend/task/MinReadyEvaluator.java)

当前它看的还是：

- `logicalSlot`

- `sizeBytes`
- `storedPath`
- `checksum`

而不是：

- `asset_id`
- `asset_version`
- `parse_status`
- `FULL_READY`
- `delta_inventory`

这意味着：

- 原计划 Week 3 方向是对的
- 但 4 周内只能追求“最小 catalog-first slice”，不能宣称完整 Metadata Catalog 已落地

### 2.4 当前 cognition / planning / validate / repair / jobs 仍压缩在一个 service 中

当前 FastAPI service 同时暴露了：

- `/cognition/goal-route`
- `/planning/pass1`
- `/cognition/passb`
- `/validate/primitive`
- `/planning/pass2`
- `/repair/proposal`
- `/cognition/final-explanation`
- `/jobs`

证据见：

- [main.py](/e:/paper_project/SAGE/Service/planning-pass1/app/main.py)

这意味着：

- 当前六层逻辑边界已经在逻辑上开始分化
- 但物理形态仍是压缩服务
- 因此 Week 4 的“能力面 contract 化”必须是薄化边界，而不是重做平台

### 2.5 当前控制层仍残留明显的领域硬编码

Java backend 里目前仍有显著的 query、case、provider 偏好硬编码与默认参数拼装。

证据见：

- control layer 里直接注入 `provider_preference / runtime_profile_preference`
- control layer 里继续写默认参数，如 `seasonality_constant / root_depth_factor / pawc_factor`

这意味着：

- 原计划 Week 4 的重点判断完全正确
- 但这项工作必须建立在 Week 2 Skill 资产化与 Week 3 Catalog slice 的基础上，否则只是“删硬编码”而不是“迁移硬编码”

## 3. 对原 4 周计划的批判性采纳结论

原计划总体上值得保留，但需要做四个关键修订。

### 3.1 保留：先做第二真实案例，而不是先做更多界面

这一点无需改动，甚至应进一步强调。

当前最大的能力缺口就是：

- case space 还没有真正成立

因此第 1 周做第二真实可执行 case，是整份计划最稳的起点。

### 3.2 修订：Week 2 不应硬承诺“2 个真实 skill 资产化”

更合理的目标应改为：

- 完整资产化一个真实主 skill：`water_yield`
- 让第二 case 通过 case registry / descriptor 接入
- 为第二 skill 预留最小资产接入骨架，但不把“第二 skill 可执行”作为 4 周内的硬 gate

原因很简单：

- 当前系统还处在单 capability 深化阶段
- 如果强行在 4 周内造第二 skill，极易出现“名义上两个 skill，实际上第二个 skill 只是薄皮壳”的失真

### 3.3 修订：Week 3 的 Catalog 目标必须压缩成最小可消费切片

更合理的说法应是：

- 让 catalog 成为一个最小但真实被消费的事实层
- 优先服务于：
  - Context Enricher
  - Pass2 materialized compilation
  - `/resume` 的 inventory merge + readiness gate

而不是追求：

- 完整 catalog 平台
- 全量 GIS 资产解析体系
- 一次性铺齐所有元数据字段与流程

### 3.4 修订：Week 4 的能力面 contract 化必须是 thin contract

原计划的方向是对的，但必须写明：

- Week 4 不做厚重能力网关重构
- 不做“新平台替换老主链”
- 不做对现有 Python service 的一刀切拆分

Week 4 的正确目标应当是：

- 让当前已有的关键边界以 contract 形式被显式定义与消费
- 先形成标准 input/output schema 与治理约束
- 再为未来能力面扩展留接口，而不是提前做重量级平台

## 4. 四周推进计划：结合当前代码的修订版

下面是在原计划基础上的修订版本。

## 4.1 第 1 周：从单 case 成功链推进到最小 case space

### 本周目标

把当前系统从：

- “只有 `annual_water_yield_gura` 一个真实可执行 case”

推进到：

- “同一 `water_yield` skill 下至少两个真实可执行 case 都能走同一条 governed 主链”

### 为什么必须第一优先级做

因为当前 clarify、case projection、registry、LLM trace 的结构都已经存在，但由于只有一个真实可执行 case，它们还不能算真正成立的 case space 能力。

### 本周要完成的功能

1. 接入第二个真实可执行 case  
   要求：
   - 不是 mock case
   - 不是 metadata-only fixture
   - 能进入真实 InVEST runtime

2. 让 clarify / case_projection / validation / execution 在两个 case 上都成立  
   不能只对 `gura` 有效。

2.1 第二 case 必须复用同一套治理主链  
   这不是“风格建议”，而是硬约束：
   - 第二 case 的接入必须复用已有的 `clarify / projection / validation / execution / result bundle / artifact / explanation` 链路
   - 不允许在任一关键阶段引入仅对第二 case 生效的新主链分支
   - 不允许为了让第二 case 尽快成功，在 Java backend 或 Python service 中新增一次性 case-specific 判断

   只有满足这一点，第二 case 的接入才代表 case space 扩展，而不是复制第一条债务。

3. 固化 case contract 的字段边界  
   至少包括：
   - `case_id`
   - `capability_key`
   - `required_roles`
   - `optional_roles`
   - `args mapping`
   - `runtime_profile`
   - `output contract`
   - `case_descriptor_version`

4. 补一组跨 case 的回归测试  
   覆盖：
   - clarify 成功
   - clarify 后 resume 成功
   - contract projection 成功
   - execution 成功
   - result bundle 落盘
   - artifact 可索引
   - final explanation 可生成

### 本周必须坚持的代码约束

- 第二 case 只能通过 registry / loader / projection / data onboarding 接入
- 不允许在 Java backend 新增新的 case-specific path assembly
- 不允许新增针对第二 case 的快捷 if-else 主链分支

### 本周验收标准

只有同时满足以下条件才算通过：

- 两个 case 都能从自然语言进入主链
- 第二 case 通过与第一个 case 相同的治理主链进入执行，而不是通过新增旁路进入执行
- 两个 case 都能形成受治理 contract
- 两个 case 都能进入真实执行
- 至少一个 case 在缺信息时能进入 `WAITING_USER`
- `/resume` 后能继续完成
- 详情页中能看到 projection / clarify / result / artifact 的完整轨迹
- 验收记录中能够明确指出：未为第二 case 新增关键阶段主链分支

### 本周最需要警惕的风险

为了让“第二个 case 尽快跑通”，偷偷写出新的 case-specific shortcut。

这类做法会让系统从“扩展 case space”退化成“复制第一条债务分支”。

## 4.2 第 2 周：把 `water_yield` 从代码知识推进到真实 skill 资产

### 本周目标

让当前主链能力第一次摆脱“主要靠代码内定义”的状态，至少完成一个真实 skill 的最小资产化闭环。

### 对原计划的关键修订

本周不建议硬性要求“至少两个真实 skill 都资产化”。

更合理的目标是：

- 完整资产化 `water_yield`
- 为未来第二 skill 准备统一资产包结构
- 如果存在自然成熟的第二 capability，再补第二个 skill skeleton
- 但不把“第二 skill 可执行”作为本周硬 gate

### 本周新增硬约束：至少一个主链节点必须强制消费 skill 资产

本周不能只交付一组资产文件，还必须明确一个运行节点在没有 skill asset 时不得继续推进。

否则很容易出现下面这种伪完成状态：

- 文件补齐了
- YAML 也存在
- 运行时依然继续读旧的 `skill_catalog.py`

那样只能算“资产存在”，不能算“资产掌权”。

因此，本周至少必须满足下面三条中的一条，且该条必须是 blocking dependency：

- route 结果必须带 `skill_id / skill_version`，且来源于已加载 skill 资产
- `passb` 的参数约束必须来自 `parameter_schema`，若 schema 缺失或绑定失败则不得继续推进
- `final explanation` 的限制性规则必须来自 `interpretation_guide`，若 guide 缺失则以结构化失败收口

如果没有任何一个主链节点在 skill asset 缺失时会阻断主链，则本周不能判定为“真资产化”。

### 本周要完成的功能

至少为 `water_yield` 补出最小资产包：

- `SKILL.md`
- `skill_profile.yaml`
- `model_mapping.yaml`
- `parameter_schema.yaml`
- `validation_policy.yaml`
- `repair_policy.yaml`
- `interpretation_guide.yaml`
- `plan_templates.yaml`
- `mcp_tools_map.yaml`

同时完成三件事：

1. 后端与 service 开始读取 skill 资产，而不是只读取代码中的 capability 配置
2. 路由结果显式返回 `skill_id / skill_version`
3. validation、repair、explanation 至少各有一个真实场景开始从 skill policy/guide 读取规则

### 本周必须面对的当前代码现实

当前 skill 知识主要还在：

- [skill_catalog.py](/e:/paper_project/SAGE/Service/planning-pass1/app/skill_catalog.py)

因此本周的关键不是“补齐文件”，而是：

- 证明这些文件已经真正进入运行路径

### 本周验收标准

- 新增一个 skill 资产版本，不需要改核心 Java 业务常量即可被识别
- `required_roles / optional_roles` 可以从资产中读出
- 参数 schema 可以参与 `args_draft` 结构约束
- `repair_policy` 至少参与一个缺参或错参场景
- `interpretation_guide` 至少参与一个 final explanation 限制性规则
- 至少一个主链节点在缺失 skill asset 或 asset binding 失败时会阻断推进，并以结构化错误收口

### 本周最需要警惕的风险

把 skill 资产化做成“把现有代码抄进 YAML”，但运行时仍然不消费这些资产。

那种做法不会减少偏离，只会制造一层文档幻觉。

## 4.3 第 3 周：把 Metadata Catalog 从概念变成最小事实源

### 本周目标

让 catalog-first 第一次成为真实主链行为，而不是停留在总体设计与认知面文档中。

### 对原计划的关键修订

本周目标应定义为：

- 构建最小 Metadata Catalog slice
- 让它被真实消费

而不是：

- 一次性建设完整 catalog 平台

### 本周要完成的功能

1. 建立最小 Metadata Catalog 结构  
   至少包含：
   - `asset_id`
   - `logical_role_candidates`
   - `file_type`
   - `crs`
   - `extent` 或 `bbox`
   - `resolution`
   - `nodata_info`
   - `source`
   - `checksum/version`
   - `availability_status`
   - `blacklist_flag`

2. Context Enricher 优先读取 catalog  
   只有 catalog 缺失或失效时，才触发受控深探测。

3. Pass2 接收 catalog facts  
   materialized compilation 不再主要依赖上传瞬时信息。

4. `/resume` 先做 inventory merge + catalog 就绪检查  
   恢复必须建立在系统事实，而不是用户口头补充。

### 本周必须面对的当前代码现实

当前真正的 ready gate 仍主要靠：

- 附件元数据
- overrides
- `waiting_context`

证据见：

- [MinReadyEvaluator.java](/e:/paper_project/SAGE/BackEnd/src/main/java/com/sage/backend/task/MinReadyEvaluator.java)

因此本周更现实的目标应是：

- 让 catalog 接管至少一条真实主链判断
- 而不是强行替换所有附件事实逻辑

### 本周验收标准

- 同一份数据二次恢复时，系统可通过 catalog 直接识别
- cognition context 能基于 catalog 完成候选筛选
- Pass2 可以输出基于 catalog 的 materialized compilation 结果
- 黑名单资产能在候选阶段被过滤

### 本周最需要警惕的风险

把 catalog 做成“另一个上传记录表”。

如果它不能被认知面、规划面、resume gate 真正消费，它就不是 Catalog，只是换名存档。

## 4.4 第 4 周：收紧治理边界，清理控制层硬编码，并做最小能力面 contract

### 本周目标

让控制层更像治理主机，而不是“懂业务语义的总控类”；同时让当前已有边界开始以 contract 形式出现。

### 为什么必须最后做

因为 Week 4 的“去硬编码”不是简单删除，而是迁移。

如果没有前 3 周的：

- 第二 case
- skill 资产
- catalog facts

那 Week 4 只能删掉现有逻辑，却没有地方可迁。

### A. 清理控制层领域硬编码

至少把以下逻辑往外迁：

- case/capability 识别硬编码
- required role 判定硬编码
- 参数默认值硬编码
- case-specific explanation 文案硬编码

迁移方向固定为：

- 路由依据 -> skill assets / case registry
- 参数结构 -> parameter schema
- repair 策略 -> repair policy
- 解释规范 -> interpretation guide

### B. 建立最小能力面 contract

本周不建设重平台，只建立一组最小 contract：

- `inspect_asset_facts`
- `validate_bindings`
- `validate_args`
- `checkpoint_resume_ack`
- `submit_job`
- `query_job_status`
- `collect_result_bundle`

之所以要补 `checkpoint_resume_ack`，是因为当前系统已经形成的真实价值不只是“能提交 job”，而是：

- `WAITING_USER`
- `/resume`
- audit
- artifact
- workspace

已经构成了一条治理闭环。

如果 Week 4 的 contract 集合只覆盖执行接口，而不覆盖恢复确认边界，那么它整理出来的仍然更像“执行接口集”，而不是“治理闭环 contract 集”。

每个 contract 至少要明确：

- input schema
- output schema
- side effect level
- caller scope
- idempotency
- cancel semantics
- audit requirement

### C. 做一轮端到端收口测试

验收对象不是“页面更漂亮”，而是：

- 2 个真实 case
- 1 个真实 skill 资产包
- 最小 catalog 进入主链
- clarify / resume 成立
- result / artifact / audit 成立
- 控制层不再直接通过硬编码判定领域语义

### 本周必须面对的当前代码现实

当前最明显的硬编码残留在：

- [GoalRouteService.java](/e:/paper_project/SAGE/BackEnd/src/main/java/com/sage/backend/task/GoalRouteService.java)
- [ExecutionContractAssembler.java](/e:/paper_project/SAGE/BackEnd/src/main/java/com/sage/backend/task/ExecutionContractAssembler.java)

因此本周的成功标志不是“写了新 contract 文件”，而是：

- 删除或迁移一段 case-specific Java 逻辑后，主链仍然成立

### 本周验收标准

- 删除一段 case-specific Java 规则后，系统仍能通过 skill/case 资产运行
- 控制层日志里能清楚区分“治理决策”与“领域决策”
- 至少 3 个能力 contract 已被真实消费
- 至少 1 个恢复相关 contract 已被真实消费，且 `/resume` 主链不依赖前端或临时旁路事实
- 架构说明可以诚实地说：当前系统已从“单纵切验证”进入“最小多 case + 单真实 skill 资产 + 最小 catalog + 最小 contract”阶段

### 本周最需要警惕的风险

把“能力面 contract 化”误做成“重写一层大网关”。

当前系统需要的是：

- thin contract
- 明确边界
- 最小真实消费

而不是：

- 新厚层
- 新平台
- 新一轮架构漂移

## 5. 横向约束：每周都必须证明“在替换旧债务”，而不只是“新增一层”

为了避免 4 周计划再次演化成“新功能叠加计划”，本阶段增加两条横向 gate。

### 5.1 横向约束 A：每周都必须有“可删除旧逻辑”的证据

每周验收都必须包含至少一条“旧逻辑被替换或可被删除”的证据，而不能只有“新增文件 / 新增接口 / 新增流程”。

最低要求如下：

- Week 1：不新增 case shortcut，且第二 case 成功不依赖新主链分支
- Week 2：至少一个旧 `skill_catalog.py` 读取路径被 skill asset loading 替换
- Week 3：至少一个旧 attachment-driven readiness 判断被 catalog slice 替换
- Week 4：至少一段 case-specific Java 逻辑可删除，且主链仍然成立

只有“能删旧逻辑”，才能证明这 4 周不是在叠层，而是在收缩偏离。

### 5.2 横向约束 B：每周验收都必须包含失效场景

每周验收不能只验证成功链，还必须验证至少一个 failure-oriented gate。

最低要求如下：

- Week 1：一个 case 缺关键 role 时必须进入 `WAITING_USER`
- Week 2：skill asset 缺 schema 或 asset binding 失败时必须阻断 `args_draft`
- Week 3：catalog miss 时必须进入 fallback probe 或 readiness fail
- Week 4：contract schema 不匹配时必须返回结构化错误码，而不是 silent fallback

只有同时验证成功链与失效链，才能证明系统成立的是治理闭环，而不是单点跑通。

## 6. 4 周结束后的签收口径

如果上述四周都按修订版完成，那么最诚实、也最接近总体设计但不过度夸大的表述应升级为：

> 当前系统已完成从单 case 验证型纵切，向“最小多 case、单真实 skill 资产化、最小 catalog 事实驱动、受治理 contract 执行框架”的过渡；但距离总体设计中的完整六层协同、完整 Skill-first 体系、完整 Metadata Catalog 与广义生态问题分析系统，仍处于早期工程化阶段。

这句话有三个优点：

- 不贬低已经取得的工程进展
- 不把“最小过渡态”说成“总体设计已落地”
- 与 [总体设计与当前实现的偏离评估.md](../架构设计/总体设计与当前实现的偏离评估.md) 的基调保持一致

## 7. 最终建议

如果把原计划与当前代码现实放在一起，我的最终建议是：

1. 第 1 周与第 4 周的方向基本正确，应坚持
2. 第 2 周必须把“2 个 skill 资产化”收紧为“1 个真实 skill 完整资产化 + 第二 skill skeleton 可选”
3. 第 3 周必须把“完整 Catalog”收紧为“最小 catalog slice 被主链真实消费”
4. 整个 4 周计划都要避免再次通过 case-specific shortcut 换取短期成功

一句话概括这份修订版：

> 未来 4 周最重要的，不是把系统讲得更完整，而是把当前最危险的四个偏离点变成真实可收缩的工程对象，并让每一个修正都能在当前代码主链中被验证。
