# 当前工作愿景、已完成进展与下一步计划

更新日期：2026-04-08

---

## 0. 文档目的

本文用于对当前阶段的工作做一次统一说明，集中回答三个问题：

1. 我们最终要做出来的东西是什么
2. 到目前为止我们已经实际完成了什么
3. 接下来最应该继续做什么

## 1. 我们要做出来的东西的愿景是什么

### 1.1 目标不是“更像聊天工具”，而是“更像可治理的分析系统”

我们要做出来的，不是一个只会回答问题的聊天机器人，也不是一个只会堆 GIS 控件的工具台。

我们要做出来的是：

> 一个以 Scene 为工作容器、以 AI 对话为入口、以地图和治理工作台为主承载、能够从分析请求进入受治理执行链并产出可追溯结果的生态问题智能分析系统。

这个系统的核心能力，不是“说得像很智能”，而是：

- 用户能用自然语言提出分析目标
- 系统能把请求约束到合法的分析类型、case、输入角色和执行合同上
- 输入不完整时，系统会进入结构化的 `WAITING_USER`
- 用户补齐输入后，系统可以沿正式 `/resume` 主链继续推进
- 执行结果、artifact、manifest、audit 和 runtime evidence 都是可追溯的
- 系统的事实层和 contract 层能够被冻结、比较、审计和演化

### 1.2 这套系统的长期工程形态

从工程上看，我们最终追求的是这样一个状态：

- 前端负责展示和交互，不承担业务裁决
- Java backend 负责状态机、治理、冻结、审计和编排
- cognition 层负责结构化 reasoning，不承担 workflow authority
- capability 层定义 skill、schema、contract 和能力元数据
- execution 层负责 job 运行和 runtime facts
- data 层负责事实、冻结对象、投影和历史记录

也就是说，系统的长期目标不是“把更多智能塞进任何一层”，而是：

> 让每一层都只做自己的事，同时让事实、治理和 contract 变得越来越稳定。

### 1.3 当前阶段最重要的愿景收缩

在当前阶段，我们不优先追求：

- 更多 skill
- 更复杂前端
- 更自由的 open-ended agent 行为

当前最重要的愿景收缩为一句话：

> 先把当前已经进入主链的 `catalog-first` 和 `contract-first` 做厚，做成可冻结、可一致性校验、可审计、可安全演化的工程边界。

## 2. 我们目前的工作已经做了哪些

### 2.1 我们已经摆脱了“单 case 样例系统”

当前 `water_yield` 已经不是单一样例闭环。

已经形成两个真实可执行 case：

- `annual_water_yield_gura`
- `annual_water_yield_gtm_national`

并且它们不是通过 Java 旁路拼出来的，而是通过：

- case registry
- case projection
- clarify
- validation
- pass2
- execution

这条同一套 governed 主链进入运行。

这意味着我们已经证明了：

- 系统不是只会跑一个写死样例
- clarify 有了真实候选空间
- case space 已经开始成立

### 2.2 我们已经完成了一个真实 skill 的最小资产化闭环

`water_yield` 已经不再主要依赖代码常量，而是拥有正式 skill 资产包。

已经完成的资产包括：

- `SKILL.md`
- `skill_profile.yaml`
- `model_mapping.yaml`
- `parameter_schema.yaml`
- `validation_policy.yaml`
- `repair_policy.yaml`
- `interpretation_guide.yaml`
- `plan_templates.yaml`
- `mcp_tools_map.yaml`

更关键的是，这些资产已经不是“文件存在”而已，而是开始进入真实运行路径：

- `PassB -> parameter_schema -> args_draft` 已成为硬 Gate
- schema 缺失或绑定失败时，不会静默 fallback 到旧 `skill_catalog.py`
- `goal-route / validation / repair / final explanation` 都已经有真实消费点
- `skill_id / skill_version` 已进入主链输出、冻结语义和任务投影

这意味着我们已经证明了：

- `water_yield` 不再只是代码知识
- skill 资产已经开始掌权

### 2.3 我们已经把最小 catalog-first slice 推进到规划链、治理链和冻结对象

当前我们还没有完整 Metadata Catalog 平台，但已经形成了真实被消费的最小 `catalog-first` slice。

已经完成的能力包括：

- `pass2` 消费 `metadata_catalog_facts`
- `MinReadyEvaluator` 基于 catalog-ready 语义决定 resume 放行
- `RepairDispatcherService` 基于 catalog facts 重算 `WAITING_USER`
- `waiting_context` 已包含 `catalog_summary`
- `detail / manifest / result` 已包含 `catalog_summary` 和 `catalog_consistency`
- `catalog_revision / catalog_fingerprint` 已进入治理语义
- frozen manifest 已持久化 `catalog_summary_json`
- task 级 `catalog snapshot` 已从“临时计算”推进到正式持久化对象

最近又继续完成了：

- catalog consistency 与 governance 视图的共享 projector / assembler
- result 路径上基于最终 runtime evidence 的 catalog coverage 计算

这意味着我们已经证明了：

- catalog 不再只是附件元数据的另一个名字
- catalog 已经进入 planning、waiting、resume、manifest、result 这些关键主链对象

### 2.4 我们已经把 contract 从“可见”推进到“真实消费”

当前最小 contract 集已经进入真实控制路径，而不是只存在于文档或 asset 文件里。

已经真实消费的 contract 包括：

- `checkpoint_resume_ack`
- `validate_bindings`
- `validate_args`
- `submit_job`
- `cancel_job`
- `query_job_status`
- `collect_result_bundle`
- `index_artifacts`
- `record_audit`

这些 contract 已经覆盖：

- 执行前
- 运行中
- 运行后
- 治理侧

同时，我们已经继续完成了：

- `contract_version / contract_fingerprint`
- frozen/current contract identity 比较
- `/resume` 的 contract mismatch 结构化收口
- `detail / manifest / result / audit` 的 contract governance 视图
- 共享的 `ContractConsistencyProjector`
- 共享的 `ContractGovernanceAssembler`

这意味着我们已经证明了：

- contract 不再只是“描述能力”
- contract 已经成为会阻断或放行主链的真实约束

### 2.5 我们已经在持续把事实计算和治理视图从 `TaskService` 中抽离

当前阶段并不是只在“加字段”，而是在持续把边界做干净。

已经完成的典型收口包括：

- `TaskCatalogSnapshotService`
- `CatalogConsistencyProjector`
- `CatalogGovernanceAssembler`
- `ContractConsistencyProjector`
- `ContractGovernanceAssembler`

这意味着我们当前在做的，不只是“功能可用”，而是：

> 把 catalog-first 和 contract-first 从零散实现推进成稳定边界。

## 3. 我们现在还没有完成什么

虽然前面的工作已经让系统发生了实质变化，但仍然不能把当前状态夸大成“总体设计已完全落地”。

当前还没完成的关键点有四类。

### 3.1 还没有完整独立的 Catalog 生命周期对象

现在的 catalog 已经进入主链，也已有 task-level snapshot，但它仍然更接近：

- attachment projection 加厚版
- task 范围内的冻结事实

而不是完整的：

- catalog parse lifecycle
- catalog ingestion lifecycle
- catalog supersede / stale lifecycle
- 独立 query contract

### 3.2 还没有完整的 Contract 演化与兼容治理

当前已经有：

- version
- fingerprint
- mismatch code
- compatibility code
- migration hint

但还没有完整做到：

- contract 版本演化策略
- 多版本并存策略
- 更细的 compatibility matrix
- 更系统化的 contract query / audit domain

### 3.3 还没有扩更多 skill / capability

当前真正资产化并进入主链掌权的，仍然主要是：

- `water_yield`

这不是缺陷，而是当前阶段有意控制的工程范围。

### 3.4 前端仍然只是最小投影，不是当前重点

前端已经接入了必要字段，但当前不是继续做 UI 包装的时候。

当前更重要的是：

- 后端事实稳定
- 治理语义稳定
- snapshot / freeze / mismatch / audit 稳定

## 4. 我们下面要做什么

### 4.1 总方向

下一阶段不优先扩广度，而优先继续做边界加厚。

也就是说，接下来最重要的不是：

- 再加一个新 skill
- 再做更多前端页面
- 再做更像 agent 的行为

而是继续把：

- `catalog-first`
- `contract-first`

这两条线做厚。

### 4.2 下一步主线 A：继续加厚 Catalog 边界

接下来最应该继续做的，是把 catalog 从“task 级 snapshot + consistency 投影”继续推进成更稳定的治理事实层。

优先项：

1. 让 `audit` 侧也进入统一的 catalog governance 视图  
   现在 `detail / manifest / result` 已有共享治理视图，下一步应该让 audit 也能直接回答：
   - 当时基于哪一版 catalog 做判断
   - 现在是否发生 drift

2. 继续把 catalog 相关事实计算从 `TaskService` 中抽离  
   例如把更多：
   - baseline/current identity 判断
   - waiting/manifest/result 的 coverage 评估
   收成更稳定的 projector/service 边界

3. 准备 catalog 专属 query / audit contract  
   让 catalog 不只作为 task read model 的附属字段存在

### 4.3 下一步主线 B：继续加厚 Contract 边界

接下来 contract 方向的重点，不是再补更多字段，而是让现有治理语义更统一、更稳定。

优先项：

1. 统一 `detail / manifest / result / audit` 的 contract drift 视图与编码  
   避免同一类 contract 漂移在不同对象里仍有不同表述习惯

2. 继续把 contract facts / governance enrichment 从 `TaskService` 中抽离  
   目标是让 `TaskService` 更接近 orchestration，而不是事实拼装中心

3. 为后续 contract query / compatibility policy 做准备  
   不是现在立刻做大平台，而是让当前的结构化语义足够稳定，未来能自然扩展

### 4.4 当前不应优先做的事

为了保持工程方向清晰，下面这些事当前不应优先：

- 扩第二个真实 skill
- 扩更多 capability 宽度
- 做更复杂的前端 UI 包装
- 把系统包装成更“智能”的聊天产品
- 提前做大而全的平台抽象

一句话说：

> 现在最重要的不是把系统做得更花，而是把系统已经进入主链的事实层和 contract 层做得更厚、更稳。

## 5. 当前阶段的工作结论

如果用一句话概括当前状态，可以这样说：

> 我们已经把系统从“单 case、代码知识、附件事实驱动、contract 只可见”的状态，推进到了“最小多 case、单真实 skill 资产掌权、catalog-first 进入规划链与治理链、contract-first 进入真实执行约束”的阶段；下一步的核心任务，不是扩展更多能力，而是继续把 catalog 和 contract 做成可冻结、可比较、可审计、可安全演化的正式边界。

这就是当前工作的真正主线。
