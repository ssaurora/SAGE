## 一、改造原则

本次改造只改后端，并遵守以下原则：

### 1. 不推翻现有 task/governance 底座

保留并继续使用现有：

* task 状态机
* waiting / resume 语义
* manifest / result / artifact / audit / catalog / contract
* traceability 与运行事实
* backend owns workflow semantics 的架构边界 

### 2. 新增一层“会话协作层”

在 task 之上新增：

* session
* message
* session context summary
* session stream
* 面向会话的 waiting/progress/result projection

### 3. 让 task 成为执行与治理底座

让 session 负责用户协作容器，task 负责：

* governed planning
* validation
* WAITING_USER
* resume
* execution
* result bundle
* audit / runs / artifacts

也就是：

> session = 用户协作容器
> task = 治理执行引擎

---

## 二、要新增的核心对象

### 1. `AnalysisSession`

新增会话主对象。

建议字段：

* `session_id`
* `title`
* `user_goal`
* `status`
* `scene_id` 或分析上下文标识
* `current_task_id`
* `latest_result_id`
* `created_at`
* `updated_at`
* `current_required_user_action`
* `session_summary`

作用：

* 承载用户一次完整分析协作过程
* 成为前端主页面的第一正式对象
* 关联当前正在推进的 task，而不是替代 task

---

### 2. `SessionMessage`

新增会话消息对象。

建议字段：

* `message_id`
* `session_id`
* `role`：`user | assistant | system`
* `message_type`
* `content`
* `created_at`
* `task_id`（可选）
* `result_id`（可选）
* `attachment_refs`（可选）
* `action_schema`（可选）
* `related_object_refs`（可选）

建议支持的 `message_type`：

* `user_goal`
* `assistant_understanding`
* `clarification_request`
* `user_clarification_answer`
* `missing_input_request`
* `upload_ack`
* `progress_update`
* `waiting_notice`
* `resume_notice`
* `result_summary`
* `failure_explanation`
* `next_step_guidance`

作用：

* 承载用户与系统的持续协作流
* 不再让前端把 event 硬解释成 message

---

### 3. `SessionContextSummary`

新增面向会话主界面的右侧上下文摘要对象。

建议字段：

* 当前目标摘要
* 当前地理对象 / case 摘要
* 当前 task 状态摘要
* 当前缺失输入摘要
* 当前推荐动作
* 当前最新结果摘要
* workbench / governance / results 快捷入口摘要

作用：

* 服务前端主会话页的“上下文侧栏”
* 避免前端从多个 task read model 手动拼装

---

## 三、要新增的后端接口

### 1. 会话创建与会话列表

#### `POST /sessions`

作用：

* 用户以自然语言创建一个分析会话
* 会话创建后，后端可立即创建初始 task，或先进入 clarifying session state

#### `GET /sessions`

作用：

* 返回会话列表
* 支持前端的全局入口页与优先处理页

---

### 2. 会话主对象读取

#### `GET /sessions/{sessionId}`

作用：

* 返回 session metadata
* 返回 session context summary
* 返回 current task summary
* 返回 latest result summary
* 返回 current required user action summary

---

### 3. 会话消息流

#### `GET /sessions/{sessionId}/messages`

作用：

* 拉取完整会话消息流

#### `POST /sessions/{sessionId}/messages`

作用：

* 用户继续输入自然语言
* 回答 clarify
* 补充要求
* 追问结果
* 请求继续推进

---

### 4. 会话级附件上传

#### `POST /sessions/{sessionId}/attachments`

作用：

* 用户在会话中上传数据
* 后端负责把附件纳入当前 task / waiting context / inventory 体系

---

### 5. 会话级流式更新

#### `GET /sessions/{sessionId}/stream`

作用：

* 以 SSE 形式推送：

  * progress update
  * waiting notice
  * result summary
  * failure explanation
  * next step guidance

说明：

* 现有 `/tasks/{taskId}/stream` 继续保留
* session stream 作为更上层的用户协作流

---

## 四、要新增的 projection / read model

## 1. `WaitingForUserProjection`

来源于现有 `waiting_context_json`，但面向会话消费。

建议包含：

* why blocked
* what user needs to provide
* acceptable answer / upload forms
* recommended actions
* resume hint
* user-facing phrasing

作用：

* 给主会话页和 Clarify 页使用
* 避免前端直接消费原始 waiting_context 细节

---

## 2. `SessionProgressProjection`

把复杂 task state 和系统内阶段，转换成可持续推送给用户的阶段反馈。

建议包含：

* current phase label
* current system action
* latest progress note
* estimated next milestone
* related task id / result id

作用：

* 支撑会话主页面中的“系统正在做什么”

---

## 3. `ResultConversationProjection`

面向主会话页展示结果摘要。

建议包含：

* result title
* short narrative
* highlights
* artifact readiness summary
* recommended next actions
* result detail deep-link target

作用：

* 让结果自然回流到会话主线，而不是只能从结果页消费

---

## 4. `SessionListSummaryProjection`

用于未来支持前端的 Session/Scene 列表页。

建议包含：

* total sessions
* needs action count
* running count
* ready results count
* priority sessions list

作用：

* 减少前端自己拼统计

---

## 五、现有 task 体系如何保留与复用

本次改造不是新增第二套执行引擎。
现有 task 体系继续保留，并承担所有治理与执行职责。

### task 继续负责

* governed planning
* pass1 / passB / validation / pass2
* WAITING_USER
* `/resume`
* execution submission
* result bundle
* artifacts
* audit
* catalog / contract views
* traceability

### session 负责

* 承接用户协作过程
* 跟踪当前协作目标
* 引用当前 active task
* 把 task 状态和结果翻译成会话流可消费消息
* 聚合当前用户应该看到的上下文

---

## 六、接口与对象之间的关系

建议采用如下关系：

* 一个 `session` 可以关联多个 `task`

  * 例如 clarify 后新 attempt，或结果后继续派生新任务
* 一个 `task` 始终隶属于某个 `session`
* 一个 `message` 可以引用 `task_id`
* result/artifact/run 仍然与 `task` 主关联
* session 通过 projection 聚合 task/result 信息，而不是复制它们

这样可以保证：

* 用户主对象是 session
* 系统执行主对象仍是 task
* traceability 不被破坏

* `POST /tasks`

* `GET /tasks/{taskId}/stream`

* `GET /tasks/{taskId}/result`

* `GET /tasks/{taskId}/contract`
* `POST /tasks/{taskId}/attachments`
* `POST /tasks/{taskId}/resume`
* `POST /tasks/{taskId}/cancel` 

### 新增会话侧接口

* 中期：前端优先使用 `/sessions`
* 高级治理页仍可继续使用 `/tasks/{taskId}` 体系

## 八、建议的实施阶段

## Phase 1：最小会话层

目标：先补正式对象，不动底层语义。

实施：

* 新增 `AnalysisSession`
* 新增 `SessionMessage`
* 新增 `/sessions`
* 新增 `/sessions/{id}`
* 新增 `/sessions/{id}/messages`
* 新增最小 `SessionContextSummary`

收益：

* 前端有正式会话主对象
* 不再只能围绕 task 做 UI

---

## Phase 2：会话消费 projection

目标：把 waiting / progress / result 变成会话可消费对象。

实施：

* 新增 `WaitingForUserProjection`
* 新增 `SessionProgressProjection`
* 新增 `ResultConversationProjection`
* 新增 `/sessions/{id}/stream`

收益：

* 前端会话页可完整消费任务推进过程
* waiting / clarify / result 不再需要前端自行硬拼

---

## Phase 3：列表与全局聚合

目标：支持产品级入口页与全局优先级页。

实施：

* 新增 `SessionListSummaryProjection`
* 新增 `priority sessions` 视图
* 视情况补 scene/session summary API

收益：

* 支撑产品前端的主入口页和优先处理逻辑
* 减少前端聚合压力

---

## 九、明确不做的事情

### 1. 不削弱现有 task/governance 底座

不因为会话层出现，就把治理能力弱化成聊天驱动逻辑。

### 2. 不让 session 替代 task

session 是协作壳，不是执行引擎。

### 3. 不把 event 当 message

event 是事实流，message 是协作流。必须分开。

### 4. 不为了前端方便随意改 task 语义

task-state authority 继续由后端严格掌控。

---

## 十、预期结果

本次改造完成后，`SAGE` 的后端形态应当从：

> 一个以 task 为第一正式对象、支持自然语言创建的治理型执行系统

升级成：

> 一个以 session 为用户协作对象、以 task 为治理执行底座的分析系统

最终，前端可以自然地做到：

* 用户持续与系统协作
* 系统在背后推进 task
* waiting / clarify / result / next step 都能自然回流到会话主线
* Governance / Result / Workbench 仍保留专业深度，但不再是唯一主入口

---

## 十一、结论

本次后端改造的本质不是“新增几个接口”，
而是正式补齐：

> **用户协作层对象 + 面向会话消费的 read model**

这样既不会破坏 `SAGE` 现有最有价值的治理底座，
又能让未来前端真正围绕“持续协作完成分析任务”来设计。
