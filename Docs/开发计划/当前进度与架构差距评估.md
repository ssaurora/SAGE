# 当前进度与架构差距评估

> 评估时间：2026-03-26  
> 评估目的：基于当前代码实现、`渐进式实施路线图` 与架构设计文档，明确项目已经做到什么程度、还缺什么、当前应如何判断所处阶段。

---

## 1. 结论先行

当前项目可以判断为：

- **Phase 1 基线基本成立**
- **尚未达到架构设计中“强治理版”的要求**
- **当前最准确的位置是：Phase 1 后期，Phase 2 起步前**

换句话说：

系统已经不是“只能演示一次的原型”，但也还不是“能力边界清晰、可规模扩展、治理机制完善”的工程化系统。

---

## 2. 当前已经做好的部分

结合当前代码、周级 E2E 脚本和已有功能，已经完成的能力主要集中在以下几条主线上。

### 2.1 主链路闭环已经存在

当前系统已经具备从任务创建到结果返回的主闭环：

- 用户提交任务
- 后端进入认知/规划/校验链路
- 缺输入时进入 `WAITING_USER`
- 用户上传附件后可 `/resume`
- 重新进入执行链路
- 执行完成后产出结果、事件、artifact、workspace 相关数据

这意味着 Phase 1 路线图中最核心的“修复闭环”已经被打通。

### 2.2 `WAITING_USER` 修复路径已经落地

从现有实现和 E2E 验证看，系统已经具备：

- 结构化表达缺失输入
- 生成 `waiting_context`
- 返回 `required_user_actions`
- 用户上传数据后重新计算可恢复条件
- `/resume` 只能在 `WAITING_USER` 状态被接受
- `/resume` 具有幂等行为

这与路线图中对 Phase 1 的第一优先项是对齐的。

### 2.3 基础 traceability 已经具备

当前后端已经存在并对外暴露：

- task detail
- result
- manifest
- runs
- artifacts
- events
- workspace 相关投影

同时，数据侧也已有对应记录对象和 Mapper，包括：

- `analysis_manifest`
- `job_record`
- `result_bundle_record`
- `artifact_index`
- `workspace_registry`
- `task_attempt`
- `repair_record`

这说明系统已经开始从“只返回结果”进入“保留执行证据与过程对象”的阶段。

### 2.4 控制面骨架已经明显形成

当前 Java backend 已经不是单纯的 Controller + Script 拼接，而是已经形成了明确的控制面骨架，包括：

- `task`
- `planning`
- `cognition`
- `repair`
- `execution`
- `validationgate`
- `event`
- `audit`
- `auth`

并且已经存在：

- `TaskService`
- `RepairDispatcherService`
- `Pass1Client`
- `Pass2Client`
- `CognitionPassBClient`
- `ValidationClient`
- `JobRuntimeClient`

这说明跨面协调关系已经成形，系统不再是单层直连。

### 2.5 认知层最小职责已经落地

当前 Service 侧已经具备最小认知/运行时服务，包括：

- `planner.py`
- `repair.py`
- `runtime.py`
- `workspace.py`
- `redis_runtime.py`
- `schemas.py`

并且 repair 相关调试能力已经打通，意味着：

- LLM 调用路径可观察
- fallback 原因可定位
- repair 输出 schema 已初步建立

这对后续进入 Phase 2 的能力化约束非常关键。

### 2.6 前端基本交互面已经成立

当前前端已经具备：

- 登录页
- 新建任务页
- 任务详情页
- 结果页
- artifacts 页

这意味着系统已经具备最小用户交互闭环，而不只是 API 工程。

## 3. 当前实现与路线图的对齐程度

### 3.1 与 Phase 0 的对齐

如果按照路线图看，Phase 0 需要的是：

- 单 Skill 最小闭环
- 两程编译保形
- 最小状态机
- 最小执行与结果返回

这一部分可以认为**已经完成**，而且已经超出。

### 3.2 与 Phase 1 的对齐

Phase 1 路线图强调的是：

- `WAITING_USER`
- Repair Dispatcher
- 修复分流
- 容器化运行
- workspace 生命周期
- `result_bundle`
- 基础审计与过程证据

从当前代码和 E2E 表现看，**这些能力大部分已经具备**。  
因此，把当前阶段判断为 **Phase 1 后期** 是合理的。

### 3.3 与 Phase 2 的对齐

Phase 2 路线图强调的是：

- 强治理
- `/resume` 背后的恢复事务
- `MIN_READY`
- Ack / Commit / Rollback
- `STATE_CORRUPTED`
- Artifact Promotion
- 更完整的规划面与能力面

这部分当前**还远未完成**。  
所以虽然系统已经有 repair loop，但还不能把它判为真正进入了路线图里的“强治理版”。

---

## 4. 当前与架构设计的主要差距

下面按六层架构逐层评估。

### 4.1 控制面：已成骨架，但远未达到强治理设计

#### 已做到

- 已有统一任务服务与状态推进主干
- 已有 repair dispatcher
- 已有 `/resume`
- 已有事件、审计、attempt、result、artifact 等对象
- 已有 auth / token / task API / result API / manifest API

#### 仍有差距

与控制面设计文档相比，当前还明显缺少：

- `MIN_READY` 作为正式恢复前门槛
- 恢复事务的 Ack / Commit / Rollback 机制
- 被控制面承认的 checkpoint/version 协调
- `STATE_CORRUPTED` 受限态治理
- 更完整的 `PRE_PROCESSING` / `ARTIFACT_PROMOTING` 等状态治理
- 强制对齐、强制回滚 checkpoint 等管理入口
- 更严格的派生状态一致性控制

#### 判断

当前控制面已经具备 **Phase 1 所需的治理主干**，但还没有达到架构设计里那种“强一致恢复控制主机”的程度。

---

### 4.2 认知面：最小闭环存在，但还没有 Skill-first 的完整形态

#### 已做到

- 已有 `goal_parse`、`skill_route`、`slot_bindings`、`args_draft`、`repair_proposal`、`final_explanation` 相关链路
- 已有 repair proposal 调用与 fallback
- 已有解释输出和调试信息

#### 仍有差距

与认知面设计相比，当前主要缺口是：

- Skill 资产体系尚未正式建立
- 认知逻辑仍较依赖当前最小实现，而不是完整 skill 约束
- memory / context 管理没有形成架构文档要求的系统能力
- repair proposal 仍偏临时实现，尚未充分能力化
- 认知输出虽然结构化，但还没有真正完全建立在 skill 资产之上

#### 判断

认知面已经具备 **受控最小实现**，但还没有进入“Skill-first 认知系统”的成熟状态。

---

### 4.3 规划面：两程接口存在，但仍偏弱化实现

#### 已做到

- 已有 `Pass1Client`
- 已有 `Pass2Client`
- 已有 `manifest` 相关对象
- 已有 `runtime_assertions`、`graph`、`planning` 相关最小链路

#### 仍有差距

与规划面设计相比，当前仍缺：

- 更完整的模板选择与模板治理机制
- `Graph Rewriter`
- `Canonicalizer`
- 更强的静态图校验
- 更完整的 graph digest / rewrite registry / planning summary 能力
- 规划面作为“工业级确定性编译器”的能力边界仍较弱

#### 判断

规划面当前处于 **保形接口已建立，但实现仍偏 Phase 0/1 弱化版** 的状态。

---

### 4.4 能力面：数据表和概念已出现，但真正的 Capability Layer 还没建起来

#### 已做到

- 已有 `CapabilityRegistryMapper`
- 已有 `ProviderRegistryMapper`
- 已有部分 contract 风格 client：
  - validation
  - planning
  - execution

#### 仍有差距

这是当前系统和总体设计差距**最大的层之一**：

- `Skill / Capability / Provider` 的正式 contract 体系还没落地
- Registry 更像是“数据准备”或“未来占位”，不是当前系统真实驱动核心
- `skill_scope / template_scope / provider_binding / contract version` 没有成为当前主链路的第一性对象
- 还没有形成完整的 validate / job / workspace / artifact / checkpoint contract 家族
- provider-first、thin gateway、version coexistence 等治理能力都还没进入当前运行主链路

#### 判断

能力面当前还停留在 **概念已出现、实现未成体系** 的阶段。  
这正是 Phase 2 最需要优先补的部分。

---

### 4.5 执行面：运行闭环存在，但工业级执行治理还未完成

#### 已做到

- 已有 job runtime client
- 已有 workspace 相关对象
- 已有取消路径
- 已有 Redis 协调
- 已有结果、artifact、workspace 过程对象
- 已完成容器化本地运行基线

#### 仍有差距

与执行面设计相比，还缺：

- 更完整的 workspace lifecycle 策略
- 更标准化的 runtime assertion push-down
- 更清晰的 preprocess / analysis 分离
- 更工业化的 isolation / provider 运行模型
- 对大规模执行与编排扩展的准备不足

#### 判断

执行面已经超出“单次脚本执行”，但还没有达到设计文档中面向大规模、强治理和高副作用操作的标准。

---

### 4.6 数据面：事实对象较齐，但恢复治理数据结构不完整

#### 已做到

- 已有 `task_state`
- 已有 `event_log`
- 已有 `audit_record`
- 已有 `job_record`
- 已有 `result_bundle_record`
- 已有 `artifact_index`
- 已有 `workspace_registry`
- 已有 `task_attempt`

#### 仍有差距

与数据面设计相比，还缺少或尚未充分进入主链路的内容包括：

- 更完整的 checkpoint / recovery 记录
- confirmation / approval / alignment 等治理表
- 更系统的 provider / contract version 冻结关系
- 更完整的恢复事务证据链
- 更清晰的 source-of-truth 与 projection 层次扩展

#### 判断

数据面已经具备 **Phase 1 的事实记录基础**，但还没有建立 **Phase 2/3 所要求的恢复与版本治理数据骨架**。

---

### 4.7 前端：可用，但还不是严格边界化展示

#### 已做到

- 已能展示任务、结果、artifact、repair 面板
- 已能发起上传、resume、查看结果与工件

#### 仍有差距

前端与架构设计相比，主要差距不是“没有页面”，而是：

- authority facts 与 advisory suggestion 的展示边界还不够清晰
- `waiting_context`、`repair_proposal`、`result_bundle` 三者的展示分层还可继续加强
- 还没有围绕 capability/skill 资产形成更强的用户可理解展示

#### 判断

前端处于 **可用态**，但还没有进入“边界清晰、认知可解释”的 Phase 2 展示形态。

## 5. 当前最明显的三个短板

如果只挑三个最关键的差距，分别是：

### 5.1 Skill / Capability 抽象尚未真正成为系统骨架

当前系统已经有 skill route、repair、validation、manifest，但这些还没有统一落到正式 capability/skill 体系里。  
这意味着系统虽然能跑，但后续扩 skill 时容易重新复制规则、复制 prompt、复制校验逻辑。

### 5.2 强治理恢复事务还没有建立

当前 `/resume` 已可用，但离控制面设计要求的：

- `MIN_READY`
- Ack
- Commit
- Rollback
- checkpoint 对齐
- `STATE_CORRUPTED`

还有明显距离。  
所以当前 repair loop 是“可工作”，但还不是“强治理恢复链”。

### 5.3 能力面与 provider 版本治理几乎还没正式开始

架构设计里非常强调：

- Registry
- scope
- provider binding
- contract version
- async capability contract

而这些目前还没有真正进入主运行面。  
这意味着系统扩展能力、替换 provider、引入多 skill 时会很快遇到工程边界问题。

---

## 6. 当前最准确的阶段判断

### 不应低估的部分

当前系统已经不是“草图级方案”，也不是“只会跑 hello world 的原型”。  
从结构对象、控制面骨架、repair loop、result/traceability、周级 E2E 来看，已经具备了真正的 Phase 1 基线。

### 不应高估的部分

当前系统也还没有达到：

- 完整 Skill-first
- 完整 capability contract
- 强治理恢复事务
- provider/version 治理
- 状态损坏治理

因此，不能把当前状态误判为“架构设计的主体都已经实现了”。

### 最终判断

当前项目最准确的判断是：

> **Phase 1 已经基本完成，Phase 2 才是接下来真正的主战场。**

---

## 7. 接下来该做什么

如果依据当前差距来排开发优先级，建议如下：

### 第一优先：建立最小 Skill / Capability Schema

这是当前最需要补的骨架。  
目标是让：

- skill 定义
- required roles / slots
- validation hints
- repair hints
- output contract

成为系统真实可消费的对象，而不是文档中的理想结构。

### 第二优先：让 waiting_context / validation / repair 对齐到 capability 事实

目标是让：

- `waiting_context`
- `required_user_actions`
- `repair_proposal`

都能基于统一 skill/capability 事实派生，而不是继续依靠散落逻辑。

### 第三优先：补控制面的强治理恢复链

重点不是“让 `/resume` 更复杂”，而是让它具备：

- 最小元数据就绪判断
- 更明确的候选变更与承认边界
- 更清晰的失败回滚语义

### 第四优先：前端拆分 authority 与 advisory 展示

这不是美化问题，而是边界问题。  
要让用户和开发者都能清楚区分：

- 系统权威状态
- LLM 建议
- 执行结果与过程证据

---

## 8. 一句话总结

当前我们已经完成了：

- Phase 0 的最小闭环
- Phase 1 的核心修复闭环与 traceability 基线

但距离架构设计里真正要求的：

- Skill-first 主体
- Capability Layer 主体
- 强治理恢复事务
- provider/version 治理
- 工业级规划与执行边界

还有一段明确且不能跳过的工程距离。

因此，当前最合理的开发判断不是“继续修补 Phase 1 脚本”，而是：

> **正式进入 Phase 2：补能力抽象、补认知约束、补恢复治理边界。**
