下面给出一版**正式的渐进式路线图**。这版不再保留讨论口吻，而是作为项目实施基线使用。其目标是：在不牺牲系统最终架构方向的前提下，以最小代价先打通内核，再逐步补齐稳定性、治理能力与工业级扩展能力。整条路线遵循现有总体设计中的基本原则：Skill-first、LLM 只承担受控认知职责、规划面采用两程编译、控制面持有唯一治理主状态机、能力面提供标准化 contract、执行面承接异步重计算。

---

# 渐进式实施路线图（正式版）

## 一、实施总目标

本路线图服务于同一条主线目标：构建一个面向生态问题分析的协同系统，使用户可从自然语言输入出发，经由 Skill 路由、结构化绑定、受控编排、异步执行、结果提取与可信解释，形成完整分析闭环。系统不以“直接调某个工具”为目标，而以“受控智能分析链条”作为交付对象。

本路线图采用“先垂直打通、再水平加固”的实施策略，分为四个阶段：

- Phase 0：最小可运行内核
    
- Phase 1：受控可用版
    
- Phase 2：强治理版
    
- Phase 3：工业级扩展版
    

各阶段以前一阶段的稳定交付为前提，不允许跳阶段堆叠复杂能力。

---

## 二、总实施原则

### 1. 结构不可偷懒，能力可以弱化

Phase 0 可以只支持单 Skill、单模板、单 provider，但不得跳过总体设计中已经明确的关键结构对象，尤其不能绕过 `logical_input_roles`、`slot_schema_view`、`slot_bindings`、`args_draft` 与两程编译节奏。规划面必须从第一阶段起就保留 Pass 1 / Pass 2 的接口形态，哪怕内部实现仍然极简。

### 2. 基础隔离前移，复杂编排后置

执行面不要求第一期就具备复杂集群编排能力，但 InVEST 与 GIS 运行环境必须尽早进入隔离运行模式。执行面设计已明确运行单元应绑定独立进程组、容器、Pod 或等价沙箱，并要求取消不能只停留在逻辑状态层。基础容器隔离应在 Phase 0 末期至 Phase 1 期间完成，K8s 级编排与规模化调度放到更后阶段。

### 3. Phase 1 的核心不是“加功能”，而是“建立修复闭环”

系统的智能感首先体现在能否结构化地区分缺数、格式不兼容、绑定失败和图结构问题。Phase 1 的首要任务是实现 `WAITING_USER` 语义、最小 repair proposal 和修复分流机制，而不是先追求更复杂的执行管线。控制面文档已将 Repair Dispatcher 与等待/恢复机制列为关键工程步骤。

### 4. 治理能力必须建立在已跑通、已稳定的闭环之上

恢复事务、`MIN_READY`、Ack/Commit/Rollback、`STATE_CORRUPTED`、Artifact Promotion、event replay、partial rerun 等能力全部属于治理增强能力，必须在基础闭环与稳定性已经成立后再引入。

## 三、Phase 0：最小可运行内核

### 3.1 阶段目标

完成从自然语言输入到一次真实 `water_yield` 分析执行结果的最短闭环。该阶段的目标不是“完整系统”，而是“可验证的系统内核”。

### 3.2 约束范围

本阶段只支持：

- 单 Skill：`water_yield`
    
- 单模板
    
- 单 provider
    
- 单分析主题
    
- 单租户
    
- 低并发
    
- 有限轮修复
    
- 单运行环境画像
    

本阶段不支持：

- 多 Skill 竞争与复杂协商
    
- 模板协商与降级协商
    
- 图重写与复杂预处理插入
    
- `/resume`
    
- `MIN_READY`
    
- Artifact Promotion
    
- partial rerun
    
- 复杂多 worker 调度
    

### 3.3 必须实现的结构链

Phase 0 虽然简化能力，但必须完整保留如下结构节奏：

`goal_parse -> skill_route -> logical_input_roles -> slot_schema_view -> slot_bindings -> args_draft -> materialized_execution_object -> job -> result_bundle -> final_explanation`

其中，规划面必须通过两程接口完成最小保形编译：

- Pass 1 输出 `selected_template`、`logical_input_roles`、`slot_schema_view`、`graph_skeleton`
    
- Pass 2 基于 `slot_bindings`、`args_draft` 与目录化事实输出 `materialized_execution_graph`、`runtime_assertions` 与最小执行对象。
    

### 3.4 本阶段各层落地要求

#### 控制面

建立最小治理骨架，包括 `TaskState`、主状态流转、`state_version`、基础事件流和幂等控制。控制面文档已明确将这一步作为工程落地第一步。

建议本阶段最小状态集合为：

- `CREATED`
    
- `COGNIZING`
    
- `PLANNING`
    
- `VALIDATING`
    
- `WAITING_USER`
    
- `QUEUED`
    
- `RUNNING`
    
- `RESULT_PROCESSING`
    
- `SUCCEEDED`
    
- `FAILED`
    
- `CANCELLED`
    

#### 认知面

只保留最小节点集：

- Intent Parser
    
- Skill Router
    
- Context Enricher
    
- Parameter Builder
    
- Validator Adapter
    
- Decision Summary Builder
    
- Skill-grounded Explainer
    

认知面负责根据规划面返回的角色与槽位进行绑定和参数草案构造，不负责模板选择、执行图生成和图级修复。

#### 规划面

必须存在，但只做最小保形两程编译。规划面不能退化成脚本直拼参数的隐式实现层。规划面应始终维持“逻辑骨架先行、实体编译后行”的制度。

#### 能力面

只落最少 contract：

- inspect
    
- validate primitive
    
- submit job
    
- query job
    
- collect / extract result
    

同时开始建立 Registry 与 Scope 意识，避免后续 capability 治理失控。能力面文档明确指出 Registry 与 Scope 是第一阶段优先项。

#### 执行面

只做 Job Runtime 骨架：

- 接受 job
    
- 启动单次分析
    
- 记录 heartbeat
    
- 返回基础状态
    
- 形成最小 result bundle
    

执行面本阶段至少要明确运行环境画像与独立执行轮次，不允许把一次运行隐式混入开发者桌面环境。

#### 数据面

建立最少事实对象，至少包括：

- `task_state`
    
- `analysis_manifest` 的最小版本
    
- `event_log`
    
- `audit_record`
    
- `job_record`
    

### 3.5 本阶段隔离要求

Phase 0 前期允许本地单机调试，但**正式演示路径必须切换到固定 Docker 镜像**。目标不是做复杂调度，而是固定运行环境，避免 InVEST / GDAL / 底层 C++ 依赖导致环境漂移。容器化在本阶段属于运行前提，不属于工业扩展项。

### 3.6 本阶段完成判据

Phase 0 完成的唯一标准是：  
用户一句自然语言能够稳定触发 `water_yield` 分析，并得到结构化结果对象与受约束解释，而不是得到脚本日志、路径列表或未治理的错误文本。总体设计明确要求解释建立在结构化结果之上，而不是把路径直接交给 LLM 猜测。

## 四、Phase 1：受控可用版

### 4.1 阶段目标

把系统从“可演示内核”升级为“可重复使用的受控系统”，重点建立结构化修复闭环、稳定运行闭环和基础工作区治理。

### 4.2 本阶段优先级

#### 第一优先：`WAITING_USER` 与结构化修复反馈

必须让系统能够清楚表达：

- 缺了哪个 slot
    
- 缺的是什么类型的数据
    
- 已有数据为什么不能接受
    
- 该补传什么
    
- 补完后可从哪一阶段恢复
    

认知面此时可以只生成简化版 `repair_proposals`，但控制面必须能够把它转换成前端可消费的等待态对象。总体设计已将 `repair_proposals` 与 `validation_result_view` 定义为认知面核心结构对象。

#### 第二优先：Repair Dispatcher

控制面正式接入跨面编排，并实现修复分流：

- 参数与绑定类问题回认知面
    
- 图结构与预处理片段类问题回规划面
    
- 缺元数据与缺输入进入等待态
    
- 明显不可继续的情况进入失败治理
    

控制面文档已把 Repair Dispatcher 作为第二步核心内容。

#### 第三优先：workspace 生命周期

执行面开始正式引入：

- workspace create
    
- workspace cleanup
    
- workspace archive
    
- workspace demolish
    
- preprocess / analysis 目录隔离
    

执行面文档明确要求 preprocess run 与 analysis run 不能共用目录命名空间，workspace 必须具备显式生命周期。

#### 第四优先：基础容器隔离常态化

每个 job 默认运行在干净容器中，形成固定输入挂载、输出回收、日志采集和取消路径。此时仍不要求复杂编排，但必须实现每 Job 的独立运行单元。

#### 第五优先：结果包与基础审计

执行面正式输出结构化 `result_bundle`，区分 `primary_outputs`、`intermediate_outputs`、`audit_artifacts` 与 `derived_outputs`，为后续结果提取和工件晋升建立基础。

### 4.3 本阶段新增 contract

能力面进入前四阶段中的前半部分，至少补齐：

- `validate_args_draft`
    
- `validate_slot_binding_primitive`
    
- `validate_input_compatibility_primitive`
    
- `submit_job_contract`
    
- `query_job_status_contract`
    
- `cancel_job_contract`
    
- `create_workspace_contract`
    
- `cleanup_workspace_contract`
    

这些 contract 对应控制面主状态机与执行面运行闭环的最小需要。

### 4.4 本阶段完成判据

Phase 1 完成后，系统应当具备：

- 能把“没给数据”和“给了但不兼容”区分开
    
- 能进入稳定的 `WAITING_USER`
    
- 能进行基础修复分流
    
- 能在容器内稳定执行
    
- 能形成结构化 `result_bundle`
    
- 能取消并真正收口运行单元
    
- 能清理 workspace，不留下隐式残留

## 五、Phase 2：强治理版

### 5.1 阶段目标

把系统从“可稳定使用”升级为“可恢复、可冻结、可回放、可审计”的治理型系统。

### 5.2 核心建设项

#### 1. 恢复与等待事务化

控制面进入第三阶段能力建设，补齐：

- `MIN_READY`
    
- `/resume`
    
- Ack / Commit / Rollback
    
- 恢复事务
    
- 状态损坏治理
    

控制面文档已明确指出，恢复不是“继续跑”，而是要在版本、输入、快照和一致性前提下重新组织事务。

#### 2. Manifest 正式冻结

总体设计要求执行前生成并冻结 `AnalysisManifest`，记录 Skill、模板、输入、参数草案、图签名、修复记录和运行约束。冻结后才进入正式分析执行。

#### 3. 规划面进入工业编译器形态

规划面逐步补齐：

- Static Graph Validator
    
- Graph Rewriter
    
- Canonicalizer
    
- Runtime Assertion Builder
    
- Graph Digest Builder
    
- Planning Summary Builder
    

同时维护独立 `PlanningState`，保证 `graph_skeleton` 与 `materialized_execution_graph` 分离保存，`planning_revision` 可回放、可审计。

#### 4. 断言下推与失败一等公民

执行面必须把断言失败显式建模为结构化失败，不得“假装成功”。断言失败后由控制面决定进入 `WAITING_USER`、`FAILED` 或修复分流。

#### 5. Checkpoint / Recovery contract

能力面补齐：

- `checkpoint_resume_ack`
    
- `force_revert_checkpoint`
    

这一步直接服务恢复事务与 `STATE_CORRUPTED` 治理。

### 5.3 本阶段完成判据

Phase 2 完成后，系统应具备：

- 用户补数后可恢复
    
- 恢复前能识别失效范围
    
- 图、参数、运行轮次三者可版本解耦
    
- 正式分析前有冻结 manifest
    
- 失败、回滚、取消和恢复均可审计、可回放
    
- 出现不一致时可进入损坏治理，而不是默默漂移

## 六、Phase 3：工业级扩展版

### 6.1 阶段目标

在治理主干已经稳定的前提下，扩展系统广度、规模与生态能力，使其进入多 Skill、多模板、多 provider、可隔离、可运维的工业运行状态。

### 6.2 主要建设方向

#### 1. 扩 Skill 与模板生态

每个 Skill 至少具备：

- `SKILL.md`
    
- `skill_profile.yaml`
    
- `model_mapping.yaml`
    
- `parameter_schema.yaml`
    
- `validation_policy.yaml`
    
- `repair_policy.yaml`
    
- `interpretation_guide.yaml`
    
- `plan_templates.yaml`
    
- `mcp_tools_map.yaml`
    

多 Skill 扩展仍必须坚持 Skill-first，不允许退化回“问题直撞工具”。

#### 2. 扩 provider 生态

能力面进入 provider 扩展阶段，通过 Registry 将 capability 解析到不同 provider，而不是不断增厚中间代理层。Skill 绑定 capability 名，而不是直接绑定底层 provider。

#### 3. 执行隔离升级

执行面从基础容器隔离升级到：

- 多 worker pool
    
- runtime profile 分层
    
- 失败隔离
    
- 资源画像调度
    
- 容器 / Pod 级运行单元
    
- partial rerun 的白名单约束
    

执行面文档已为 worker 类型、runtime profile、workspace 生命周期与强取消定义了基础模型。

#### 4. Artifact Promotion 与终局治理完整化

分析成功后，工件晋升、校验、归档、终局销毁和保留策略形成完整闭环。执行面不拥有最终晋升裁决，但必须提供 promotion job、checksum、verify 支持，并在控制面指令下完成终局清理。

#### 5. 数据生命周期治理

进入规模化后，数据面补齐：

- archive / retention policy
    
- bucket / prefix 约定
    
- workspace 与 artifact 生命周期规则
    
- 事件回放与审计复盘能力
    

### 6.3 本阶段完成判据

Phase 3 完成后，系统应当能够支撑：

- 多 Skill 并存
    
- 多模板演进
    
- 多 provider 接入
    
- 多运行环境画像
    
- 可观测、可治理、可清理的长时运行
    
- 更严格的审计、归档和复盘要求

## 七、各阶段硬红线

### Phase 0 红线

- 不允许把规划面省略成脚本拼参数
    
- 不允许让认知面直接产出最终执行参数并直撞执行
    
- 不允许正式演示仍依赖开发者本地裸环境
    

### Phase 1 红线

- 不允许没有 `WAITING_USER` 结构化语义就宣称“可用”
    
- 不允许 cancel 只有状态变化，没有物理终止
    
- 不允许 workspace 无生命周期治理
    

### Phase 2 红线

- 不允许没有冻结与恢复事务就引入复杂复用
    
- 不允许没有审计 revision 就做图重写与 checkpoint 回退
    
- 不允许跨层状态不一致而无损坏治理
    

### Phase 3 红线

- 不允许多 Skill 扩展破坏 Skill-first 原则
    
- 不允许 provider 扩展演化为厚重代理中台
    
- 不允许把执行面变成策略裁决层
    

---

## 八、正式实施顺序

本项目正式实施顺序确定如下：

1. **先做 Phase 0**  
    围绕 `water_yield` 建立最小闭环，验证结构链正确。
    
2. **再做 Phase 1**  
    以 `WAITING_USER`、Repair Dispatcher、容器常态化、workspace 生命周期和 `result_bundle` 为重点，验证系统稳定性。
    
3. **再做 Phase 2**  
    以恢复事务、Manifest 冻结、规划面工业编译器增强、断言下推和损坏治理为重点，验证系统治理能力。
    
4. **最后做 Phase 3**  
    在主闭环、稳定性和治理性都成立的基础上，再扩多 Skill、多 provider、多运行环境和工业运维能力。
    

---

## 九、路线图结论

这条渐进式路线图的核心，不是把复杂能力后置，而是把**正确结构前置、复杂治理分期、环境隔离提前、修复闭环前置**。  
它保证第一期不是“快糙猛的脚本原型”，第二期不是“只会跑不会解释的工具壳”，第三期不是“有恢复名词没有事务语义的伪治理系统”，第四期也不会因为前期结构欠债而被迫整体返工。

如果你认可这版正式路线图，下一步就应该把 **Phase 0 与 Phase 1** 直接拆成研发任务表。
