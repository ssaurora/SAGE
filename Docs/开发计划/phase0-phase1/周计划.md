# 第 1 周：把治理骨架和运行骨架立起来

## 周目标

建立系统最小骨架，让任务对象、状态流和运行环境不再是散的。

## 本周要完成的功能

### 1. 控制面核心任务对象

完成：

- `task_state`
    
- `event_log`
    
- `audit_record`
    
- 最小状态机骨架
    
- `POST /tasks`
    
- `GET /tasks/{taskId}`
    
- `GET /tasks/{taskId}/events`
    

### 2. 规划面 Pass 1 雏形

完成：

- 固定 `water_yield` 模板
    
- 返回 `selected_template`
    
- 返回 `logical_input_roles`
    
- 返回 `slot_schema_view`
    
- 返回 `graph_skeleton`
    

### 3. 执行环境镜像

完成：

- 一份固定 Docker 镜像
    
- 容器内固定 Python / InVEST / GIS 依赖
    
- 固定输入、输出、日志目录规范
    

### 4. 前端任务页骨架

完成：

- 输入页
    
- 任务详情页骨架
    
- 状态展示
    
- 事件列表展示
    

## 使用技术栈

- **Spring Boot / Java**：控制面 API、状态机、事件记录
    
- **PostgreSQL**：`task_state`、`event_log`、`audit_record`
    
- **Python**：规划面 Pass 1
    
- **Pydantic**：Pass 1 输出对象 schema
    
- **Docker**：执行环境固定
    
- **React / Next.js / TypeScript**：前端页面骨架
    

## 验收标准

本周结束必须满足：

1. 用户可以创建一个任务
    
2. 后端能生成唯一 `task_id` 并落库
    
3. 任务状态至少能从 `CREATED` 推进到 `COGNIZING / PLANNING`
    
4. 前端可以看到状态与事件流
    
5. Pass 1 不再是“口头约定”，而是真实返回 `logical_input_roles` 和 `slot_schema_view`
    
6. Docker 镜像可以独立启动，不依赖开发者本地裸 Python 环境

# 第 2 周：把 Pass 1 接到认知绑定，形成可验证输入链

## 周目标

让系统从“有骨架”变成“能围绕角色和槽位组织输入”。

## 本周要完成的功能

### 1. Cognition Pass B

完成：

- 读取 `logical_input_roles`
    
- 生成 `slot_bindings`
    
- 生成 `args_draft`
    
- 返回 `decision_summary`
    

### 2. 最小 Validation Gate

完成：

- 校验 `slot_bindings`
    
- 校验 `args_draft`
    
- 校验失败时先进入 `WAITING_USER` 或 `FAILED`
    

### 3. 控制面阶段编排

完成：

- `POST /tasks` 后串起：
    
    - Planning Pass 1
        
    - Cognition Pass B
        
    - Validation Gate
        

### 4. 前端详情页增强

完成：

- 展示 `slot_bindings` 摘要
    
- 展示 `args_draft` 摘要
    
- 展示验证结果摘要
    

## 使用技术栈

- **Spring Boot / Java**：控制面编排与 Validation Gate
    
- **Python**：认知面 Pass B
    
- **LLM 调用层**：生成绑定候选与参数草案
    
- **Pydantic**：`slot_bindings`、`args_draft` schema
    
- **PostgreSQL**：保存阶段结果摘要
    
- **React / Next.js / TypeScript**：详情页增强
    

## 验收标准

本周结束必须满足：

1. Pass 1 的输出被真实用于 Pass B，而不是重新手写参数
    
2. 系统可以生成结构化 `slot_bindings`
    
3. 系统可以生成结构化 `args_draft`
    
4. Validation Gate 在控制面统一执行，而不是某个 Python 节点自己决定是否继续
    
5. 前端能看到“当前绑定了什么、参数草案是什么、校验是否通过”
    
6. 失败不再只是异常堆栈，而是有明确验证失败摘要

# 第 3 周：把 Planning Pass 2 和 Job Runtime 打通

## 周目标

完成从“绑定和草案”到“可执行对象”的转换，并把 analysis job 真正提交起来。

## 本周要完成的功能

### 1. Planning Pass 2

完成：

- 输入 `slot_bindings + args_draft + graph_skeleton`
    
- 输出 `materialized_execution_graph`
    
- 输出最小 `runtime_assertions`
    
- 输出 `planning_summary`
    

### 2. `job_record`

完成：

- 建表
    
- 记录 job 生命周期
    
- 记录 `job_id`
    
- 记录 job 状态
    

### 3. Analysis Job 提交

完成：

- 控制面提交 job
    
- Python Worker 接受 job
    
- job 在 Docker 中运行
    
- heartbeat 回传
    

### 4. 最小结果对象

完成：

- 主要输出
    
- 日志路径
    
- 关键指标摘要
    
- 失败信息
    

## 使用技术栈

- **Python**：Planning Pass 2、Worker
    
- **Pydantic**：执行对象、断言对象
    
- **NetworkX 或轻量图结构表达**：执行图对象
    
- **Spring Boot / Java**：提交 job、记录状态
    
- **PostgreSQL**：`job_record`
    
- **Docker**：analysis job 运行单元
    

## 验收标准

本周结束必须满足：

1. Pass 2 真实存在，并能输出 `materialized_execution_graph`
    
2. 控制面拿到的是“执行对象”，不是直接去拼脚本参数
    
3. 至少有一个 `water_yield` job 能被成功提交
    
4. job 状态能从 `ACCEPTED` 走到 `RUNNING` 再到终态
    
5. 后端可追踪 `task_id -> job_id`
    
6. 结果至少形成最小结构化对象，而不是只留下日志文件

# 第 4 周：把 Phase 0 正式跑通

## 周目标

拿到一条完整可演示链：自然语言输入到真实 `water_yield` 结果。

## 本周要完成的功能

### 1. Result Bundle 雏形

完成：

- `primary_outputs`
    
- `logs`
    
- `failure_info`
    
- 关键指标摘要
    

### 2. Final Explanation

完成：

- 基于结构化结果生成解释
    
- 不允许直接根据底层日志或路径自由生成结论
    

### 3. Cancel 最小闭环

完成：

- `/cancel`
    
- Worker 响应 cancel
    
- 停掉主进程或容器
    
- 更新任务状态
    

### 4. 前端结果页

完成：

- 展示结果摘要
    
- 展示输出清单
    
- 展示解释
    
- 展示失败信息
    

## 使用技术栈

- **Python**：result bundle builder、解释输入整理
    
- **LLM 调用层**：final explanation
    
- **Spring Boot / Java**：结果接口、cancel 接口
    
- **PostgreSQL**：结果摘要、失败信息
    
- **Docker**：cancel 真终止
    
- **React / Next.js / TypeScript**：结果页
    

## 验收标准

本周结束必须满足：

1. 用户一句自然语言可以触发完整 `water_yield` 流程
    
2. 系统完成：Pass 1 → Pass B → Pass 2 → validate → job → result → explanation
    
3. 结果页展示的是结构化结果与解释，而不是目录路径堆
    
4. cancel 可以真正终止运行中的 job
    
5. 演示链路完全运行在固定 Docker 环境中
    
6. 到这周结束，**Phase 0 视为完成**

# 第 5 周：把 WAITING_USER 和修复分流做出来

## 周目标

让系统不只是“跑得通”，还要“知道为什么跑不通，并且告诉用户怎么办”。

## 本周要完成的功能

### 1. `WAITING_USER` 结构化对象

完成：

- `missing_slots`
    
- `invalid_bindings`
    
- `required_user_actions`
    
- `resume_hint`
    

### 2. Repair Dispatcher

完成：

- 参数/绑定问题 → 回认知面
    
- 图结构问题 → 回规划面
    
- 缺数问题 → `WAITING_USER`
    
- 系统问题 → `WAITING_SYSTEM`
    
- 致命问题 → `FAILED`
    

### 3. 前端补数面板

完成：

- 展示缺失 slot
    
- 展示问题原因
    
- 提示用户补什么
    
- 提供上传入口
    

### 4. `/resume` 基础入口

完成：

- 用户补数后可以触发恢复
    
- 恢复前做最小检查
    
- 成功后重新进入编排链
    

## 使用技术栈

- **Spring Boot / Java**：Repair Dispatcher、等待态对象、`/resume`
    
- **Python 认知面**：repair proposal / waiting reason 生成
    
- **Python 规划面**：图结构类错误返回
    
- **PostgreSQL**：等待态持久化
    
- **React / Next.js / TypeScript**：补数面板、恢复入口
    
- **对象存储 / 上传服务**：补传文件
    

## 验收标准

本周结束必须满足：

1. 系统能明确区分“缺数”和“已有数据不兼容”
    
2. 任务进入 `WAITING_USER` 时，前端能显示结构化缺口信息
    
3. 用户知道要补什么，而不是只能看到失败提示
    
4. `/resume` 可以被调用并重新进入执行链
    
5. 常见失败不再都落成一个统一失败出口
    
6. 到这周结束，**Phase 1 的修复闭环基础成立**

# 第 6 周：把 workspace、工件索引和运行态治理站稳

## 周目标

补齐 Phase 1 的稳定性基础，让系统可以被重复使用，而不会留下环境、目录和工件层面的混乱。

## 本周要完成的功能

### 1. `workspace_registry`

完成：

- 建表
    
- 记录 workspace 与 task/job/run revision 的关系
    

### 2. workspace 生命周期

完成：

- create
    
- cleanup
    
- archive
    
- demolish
    

### 3. `result_bundle_record`

完成：

- bundle 摘要落库
    
- 支持任务与结果对象关联
    

### 4. `artifact_index`

完成：

- 工件索引
    
- 工件分类
    
- 存储位置引用
    
- 与 workspace / run revision 建立追溯关系
    

### 5. Redis 运行态协调

完成：

- heartbeat
    
- lease
    
- cancel token
    

### 6. Capability / Provider Registry 最小版

完成：

- `capability_registry`
    
- `provider_registry`
    
- 注册：
    
    - validate
        
    - submit job
        
    - query job
        
    - cancel
        
    - workspace
        
    - result
        

## 使用技术栈

- **PostgreSQL**：`workspace_registry`、`result_bundle_record`、`artifact_index`、registry 表
    
- **Python Worker**：workspace 生命周期实现
    
- **Spring Boot / Java**：registry 查询与调用路由
    
- **Redis**：heartbeat / lease / cancel token
    
- **Docker**
    
- **对象存储**
    

## 验收标准

本周结束必须满足：

1. 任一 workspace 都能追溯到 task/job/run revision
    
2. 运行结束后临时目录可清理，不再堆积匿名文件夹
    
3. `result_bundle` 不只是内存对象，而是有稳定记录
    
4. 工件可通过 `artifact_index` 被追踪，而不是直接写死路径
    
5. Redis 只承担运行态协调，不承担主事实存储
    
6. 能力调用开始走 registry，而不是全靠硬编码
    
7. 到这周结束，**Phase 1 视为完成**
